说明：

- 项目默认通过 `nginx` 对外监听 `80` 端口，后端 Gunicorn 仅绑定 `127.0.0.1:8000`。
- 阿里云安全组只需要放行 `80/TCP`；`8000` 不需要暴露到公网。
- `start_aliyun.sh` 会自动跳过已安装的系统依赖、已存在的虚拟环境和未变更的 Python 依赖。
- `start_aliyun.sh` 会自动写入 `WEB_CONCURRENCY=4`、`GUNICORN_TIMEOUT=120`、`GUNICORN_KEEPALIVE=75`、`GUNICORN_GRACEFUL_TIMEOUT=30`；4 核机器可从 2 到 4 worker 开始，8 核机器可从 4 worker 开始。
- `start_aliyun.sh` 写入的 systemd 服务会使用 Gunicorn + UvicornWorker，并设置 `LimitNOFILE=1048576`。
- `start_aliyun.sh` 写入的 Nginx 配置会保留 SSE 所需的 `proxy_buffering off`、`proxy_request_buffering off`、`proxy_read_timeout 600s`、`proxy_send_timeout 600s`。
- `start_aliyun.sh` 会自动写入上游连接池建议值：`REQUEST_TIMEOUT_MS=60000`、`UPSTREAM_MAX_CONNECTIONS=1200`、`UPSTREAM_MAX_KEEPALIVE_CONNECTIONS=300`、`UPSTREAM_POOL_TIMEOUT_S=10`。
- `start_aliyun.sh` 会自动写入流式超时建议值：`STREAM_CONNECT_TIMEOUT_SECONDS=10`、`STREAM_FIRST_TOKEN_TIMEOUT_SECONDS=60`、`STREAM_IDLE_TIMEOUT_SECONDS=120`、`STREAM_MAX_DURATION_SECONDS=600`。
- 中转站容量可在后台配置最大活跃请求、最大流式请求、QPS、失败率上限和首 Token 超时；当前阶段只做低阈值功能验证，不做真实高并发压测。
- `start_aliyun.sh` 会安装并启动 `redis-server`，默认写入 `REDIS_URL=redis://127.0.0.1:6379/0`，并通过 Redis 处理全局、API Key、账户和中转站级并发计数与 QPS/RPM 限流。
- API Key 鉴权缓存同样使用 Redis，默认 `API_KEY_AUTH_CACHE_TTL_SECONDS=60`；该缓存只保存 SHA256 哈希对应的鉴权快照，不保存明文 Key，生产变更 API Key 或账户状态后会主动失效。
- 日志、Token 回填和计费由后台任务补齐；生产建议保持完整 payload 和流式响应持久化关闭，仅在排障时临时开启，并通过 `max_logged_body_bytes` 限制体积。
- 并发上限与流式超时可通过 `.env` 初始化，也可在后台“系统设置”中调整；生产验证时可以临时把 `GLOBAL_MAX_ACTIVE_REQUESTS` 或 `GLOBAL_MAX_ACTIVE_STREAMS` 调为 1 或 2 做少量人工验证，禁止在 provider 不足时做真实高并发压测。
- `/ready`、`/metrics` 和后台 `/api/metrics/system` 会输出 Redis、数据库、连接池、活跃请求、流式活跃、provider 失败率、429/5xx、后台任务积压、Token/计费失败等核心指标；告警中心会自动写入对应监控告警。
- `start_aliyun.sh` 会自动补齐生产环境密钥，并在 `EXTERNAL_BASE_URL` 为空时尝试写入公网 IP。
- 部署脚本执行完成后会自动校验 `http://127.0.0.1:8000/login`，并在可识别公网地址时校验公网 `/login`。
- 若需要文档页固定显示公网接入地址，请在 `.env` 中设置 `EXTERNAL_BASE_URL=http://你的公网IP` 或正式域名。

如果你怀疑环境、依赖、.env、systemd 配置有变化，或者想“重新执行整套部署”，可以直接再跑一次脚本；这个脚本本身就是按“可重复执行”写的，已有内容会尽量跳过：

cd /opt/auto-gpt
sudo ./start_aliyun.sh

按当前项目规范，标准建议是：

日常重启服务：sudo systemctl restart aotu-gpt
改了反向代理配置时：再执行 sudo systemctl restart nginx
改了依赖、部署配置或怀疑环境异常时：sudo ./start_aliyun.sh

但有两个前提：

git status -sb 不能有本地代码改动。
如果还有 M start_aliyun.sh，先执行：
cd /opt/auto-gpt
cp -a start_aliyun.sh /root/start_aliyun.sh.bak.$(date +%Y%m%d%H%M%S)
git restore -- start_aliyun.sh
sudo bash update_aliyun.sh

数据库结构变更后，如果更新后报 no such column 之类错误，再执行一次：
cd /opt/auto-gpt
sudo env RUN_DB_INIT=1 bash start_aliyun.sh

这个会先备份 PostgreSQL 数据库，再做初始化/补字段。


# 推荐更新命令
cd /opt/auto-gpt

# 1. 先看是否有本地代码改动
git status -sb

# 2. 如果只看到 start_aliyun.sh 被改过，先备份再恢复它
cp -a start_aliyun.sh /root/start_aliyun.sh.bak.$(date +%Y%m%d%H%M%S)
git restore -- start_aliyun.sh

# 3. 备份运行配置和本地数据
mkdir -p /root/aotu-gpt-backups
cp -a .env /root/aotu-gpt-backups/.env.$(date +%Y%m%d%H%M%S)
cp -a data /root/aotu-gpt-backups/data.$(date +%Y%m%d%H%M%S)

# 4. 如果你用 PostgreSQL，先手动备份数据库
source .env
pg_dump -Fc -f /root/aotu-gpt-backups/aotu-gpt-db.$(date +%Y%m%d%H%M%S).dump "${DATABASE_URL/postgresql+psycopg/postgresql}"

# 5. 更新代码和服务
sudo bash update_aliyun.sh
