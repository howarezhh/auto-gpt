# 项目全局规范

前端设计规范：如果任务与前端设计需求有关，请务必先阅读 前端设计规范.md 文件完整内容，如果该项目根目录下面没有该文件，则无需遵守该规范，也不用新建文件！！！！！！！！。

## 1. 规范文件

- 项目根目录必须且仅允许维护一份 `项目全局规范.md` 作为唯一有效的全局规范正文文件。
- 项目根目录必须维护 `《项目全局规范-变更记录.md》` 作为全局规范变更记录唯一落地文件。
- 项目根目录下的问题经验沉淀文件必须为 `项目问题及解决方法记录.md`。

## 2. Python 环境

- 若任务需要使用 Python 环境，必须优先使用项目根目录下已有虚拟环境。
- 若项目根目录下不存在可用虚拟环境，才允许在项目根目录下新建虚拟环境。
- 若需要安装 Python 依赖，必须先指定国内镜像源，再执行安装。

## 3. 变更控制

- 执行任何任务前，必须先检查当前需求是否与本文件冲突。
- 若当前需求与本文件或已确认的历史统一标准冲突，必须先暂停执行，说明冲突点，并等待用户确认后再继续。
- 若用户已明确确认可在某一连续实施任务内由执行者自行裁定局部规范冲突方案，则允许仅在该次连续实施范围内按“最小必要变更、保持可追踪、优先不破坏既有正式对外边界”的原则继续执行；若后续需要把该裁定方式长期化，必须再单独修订本文件。
- 只有当新规则具备长期复用价值、可统一执行、可减少返工或歧义时，才允许写入本文件。
- 规范更新必须最小化、可执行、可检查、可追踪，禁止写入一次性任务要求或临时偏好。



## 4. 实施约束

- 当前任务新增或修改的实现，必须优先复用项目现有目录结构、命名方式、接口风格和数据组织方式。
- 若发现历史实现与现行规范不一致，必须先指出不一致；未获明确要求时，仅对当前任务相关范围做最小必要修改，不擅自扩展为全项目重构。
- 若修改 `app/static/` 下会影响浏览器执行结果的静态资源，必须同步更新 `app/templates/base.html` 中对应资源的版本参数，避免页面结构与缓存脚本或样式不一致。
- 面向后台运维或业务用户的界面文案、下拉选项、表格展示值、卡片标题在不涉及接口路径、字段名、变量名、模型名、错误码等技术标识时，必须优先使用中文显示；若底层仍使用英文枚举值，界面层必须做中文标签映射，禁止仅向用户显示英文枚举值。
- 全站前端页面在同一页面内的卡面容器必须采用无阴影的一体化视觉；禁止继续通过阴影把 `panel`、`metric`、`hero`、`card` 等内容块做成彼此割裂的悬浮卡片。
- 全站前端页面中用于布尔设置启停的交互控件必须优先使用滑块式开关，不得继续直接向用户裸露浏览器默认勾选框作为主要视觉形态。
- 全站前端页面中的按钮、图标按钮、导航按钮及其他按钮式交互控件必须默认不显示可见方框或边框；仅允许在 `hover`、`focus-visible` 等指向态显现提示性边界，禁止默认做成边界分明的方框按钮。
- 后台管理端与用户业务端的全局主导航在桌面端必须放置于左侧侧边栏，顶部区域仅允许承载轻量工具、账号、主题切换或页面上下文信息；移动端必须提供可折叠导航入口，禁止把全量主导航长期铺在页面顶部。
- 后台管理端与用户业务端的数据密集页面外壳必须尽量占满可用屏幕宽度，仅保留统一安全边距；表格、统计卡片、筛选区和操作面板必须优先铺满主内容区。仅正文阅读区、登录/注册卡片、弹窗和代码示例等需要控制可读性的内容允许单独限宽。
- 后台管理页与用户业务页的筛选区必须优先使用同排等宽栅格布局；关键词、状态、归属、分页数量等同层级筛选项在桌面与平板宽度足够时必须尽量排在同一行且控件宽度、高度保持一致，仅在移动端或容器宽度不足时允许自然换行或单列展示，禁止无业务必要地把每个筛选项做成独占整行的堆叠列表。
- 后台管理页与用户业务页禁止继续放置脱离实时数据、表单提交或实际操作入口的纯说明性前端内容；对象管理流程、建议阅读顺序、治理提醒、风险提示、判断原则、下一步动作等静态流程卡片、建议卡片、说明卡片必须移除。确有必要保留的操作说明，仅允许紧贴具体控件以内联文案、占位提示、Tooltip、空状态或结果反馈形式出现，不得再单独做成说明性卡片或侧栏说明块。
- 后台管理页与用户业务页禁止放置独立快捷入口卡片、快速跳转卡片或常用动作卡片；跨页面跳转必须优先通过全局导航、页面主操作按钮、锚点导航或贴近业务对象的必要操作呈现，禁止以 `快捷入口`、`快速跳转`、`常用动作` 等独立卡片堆放链接。
- 对外统一代理入口必须使用 `/v1/*` 路径并采用外部 Bearer API Key 鉴权；内部管理或调试调用禁止直接复用对外 `/v1/*` 入口。
- 人类用户登录态必须与对外 API Key 鉴权分离：管理后台端与用户端统一使用基于服务端 Session 的浏览器登录态，对外 `/v1/*` 入口禁止以人类用户登录态替代 Bearer API Key。
- 前端页面路由必须维护统一访问控制规则；受保护页面既必须保留服务端 Session 校验，也必须在前端壳式导航层按登录态与角色做集中拦截，禁止各页面各自散落判断。
- 未登录访问受保护页面时，前后端必须统一跳转到 `/login`，并通过 `next` 参数保留原目标路径；登录成功后仅允许回跳到当前角色可访问的页面。
- 已登录但角色不匹配时，前端路由守卫与服务端页面路由必须统一重定向到该角色的默认首页；管理员默认首页为 `/`，普通用户默认首页为 `/user`。
- 内部 playground 或后台调试能力必须通过 `/api/playground/*` 等内部管理接口接入，不得要求浏览器直接持有外部 API Key。
- 浏览器侧的内部多模态素材上传、预处理和临时资源生成能力必须通过 `/api/playground/assets/*`、`/api/user/assets/*` 等内部 Session 接口接入，禁止把此类浏览器辅助能力直接暴露到对外 `/v1/*` 正式代理入口。
- 对外 `/v1/*` 入口禁止接收仅供内部调试使用的强制路由参数或指定 provider 头；“指定某个具体中转站测试”只允许出现在 `/api/playground/*` 链路。
- 对外响应式接口必须使用标准 `/v1/responses` 路径，禁止新增或沿用非标准 `/v1/response` 路径。
- API Key 相关请求日志必须继续复用 `request_logs` 扩展字段落库，禁止新建第二套重复的请求日志主表。
- 模型调用日志必须在 `request_logs` 中统一落库 `session_id`、`user_account_id`、`user_account_name`、`http_method`、`reasoning_level`、`attempt_count`、`ttfb_ms`、`duration_ms`、`tps`、`cache_read_tokens`、`cache_write_tokens`、`billing_multiplier`、`channel_price_input_per_1k`、`channel_price_output_per_1k` 等核心分析字段，禁止只在前端临时拼装。
- 对外代理主链路必须只同步写入 `request_logs` 最小核心字段；Token 估算、费用计算、余额扣减和账单记录必须通过可重试且幂等的后台任务完成，禁止在请求主链路执行大文本 Token 计算或重复扣费。
- `request_logs` 的 `request_body_json`、`response_body_json`、`response_text` 等大字段必须受 payload 日志开关、流式响应持久化开关和最大记录字节数控制；高并发生产默认禁止保存完整成功请求/响应 payload。
- 后台日志中心与统计看板需要提供可导出的结构化结果时，必须优先复用 `request_logs` 的统一筛选口径生成 CSV 或时间维度聚合结果，禁止在前端基于当前页临时数据拼接“伪导出”。
- 管理端日志中心必须支持按用户筛选全量日志；普通用户端日志页必须只展示当前账号名下 API Key 产生的模型调用日志，并强制排除健康检查、后台测试和其他非用户调用日志。
- 管理端必须提供可检索的操作审计日志页面；用户、API Key、模型、系统设置、账户调账等关键后台写操作必须统一记录操作者、对象、摘要、时间与必要变更详情。
- 系统设置必须支持配置 `request_logs` 与管理端审计日志的保留天数，并由后台定时任务按统一配置自动清理；当保留天数为 `0` 时，表示禁用该类日志的自动清理。
- API Key 专属路由上下文必须至少包含 `route_mode`、`default_provider_id`、`manual_allow_fallback`、`allowed_provider_ids` 四项约束。
- 人类用户体系必须至少区分 `admin` 与 `user` 两类角色；默认关闭公开注册，管理员必须能在后台切换是否允许普通用户公开注册，并始终保留管理员后台创建账号的能力。
- 外部 Bearer API Key 鉴权除密钥本身校验外，必须继续支持账户级聚合控制；管理员必须能够按账户配置调用次数、Token、金额的总额度以及日/月配额，并在鉴权链路统一生效。
- 账户级日配额与月配额必须使用统一时区口径统计；当前项目默认按 UTC 自然日与自然月计算，禁止在不同模块混用不同时间窗口。
- 平台 API Key 管理必须同时支持自动生成与管理员自定义录入；自定义密钥必须按统一规则校验长度、前缀和字符集，且后台列表必须支持查看与一键复制当前可回显密钥。
- 管理端 API Key 管理页必须支持服务端分页、服务端筛选、后端导出以及批量启用、批量禁用、批量删除、批量渠道授权更新；禁止仅基于当前前端已加载页数据做本地假分页或伪批量操作。
- 普通用户端必须支持对当前账号名下 API Key 执行新增、编辑、启停、删除和明文复制；用户侧自助管理仅允许作用于本人名下密钥，禁止修改密钥归属账号以及账户级额度、余额、冻结金额等管理员治理字段。
- 普通用户端必须提供 API Key 详情页，并支持一键轮换当前账号名下 API Key；轮换后旧密钥必须立即失效，详情页必须仍可提供最新明文复制能力。
- 普通用户端必须支持日志 CSV 导出、账单 CSV 导出和接入自检入口；接入自检至少需要覆盖当前账号名下 API Key 状态、授权中转站、目标模型与可命中路由候选的可视化检查。
- 需要支持后续回显或复制的 API Key 明文禁止只做不可逆哈希存储；必须使用受控密钥进行加密存储。历史未存明文的旧密钥不得伪造回显，必须明确提示管理员通过编辑重置。
- 后台中转站价格录入、编辑和展示界面必须统一使用“每百万 token”作为用户可见单位；底层数据库与计费链路若继续复用 `*_per_1k` 字段，仅允许在界面或接口适配层做双向换算，禁止直接改写历史存量数据口径。
- 后台中转站新增与编辑流程必须支持基于当前中转站配置请求上游 `/models` 拉取可用模型列表，并支持把选中的一个、多个或全部上游模型一键导入当前中转站模型配置。
- 后台必须提供独立的模型聚合配置能力；模型基础价格、速度标签、全局启用状态和可用渠道必须以“模型”为主视角统一维护，禁止仅依赖中转站页面分散维护。
- 同一模型的渠道价格必须采用“模型基础价 × 渠道倍率”口径；倍率定义为单个渠道对单个模型的价格倍率，渠道最终价格必须按该公式换算。
- 模型平均倍率必须作为只读派生指标由已绑定渠道倍率自动计算，禁止作为独立可编辑字段；若需要改变平均倍率，必须通过调整对应渠道绑定关系或单渠道倍率实现。
- 用户端必须提供可用模型页面，并且仅允许展示当前账号已授权范围内可访问模型的价格、速度标签和可用渠道，禁止向普通用户暴露后台全量模型池。
- 普通用户端必须提供可访问的 `使用文档` 页面；该页面允许复用统一外部接入文档模板，但用户端导航中必须存在独立入口，且内容口径必须与管理端 `使用文档` 页面保持一致。
- 账户级余额可视化必须至少展示账户总余额、冻结金额、可用余额、已消费金额、累计充值金额和消费明细；账户余额必须以 `user_accounts` 共享金额池为唯一口径，同一用户名下所有 API Key 共用同一账户余额，禁止继续以名下 API Key 余额求和冒充账户余额。
- 管理端用户管理必须支持批量账户额度调整；批量调整至少需要覆盖调用次数、Token、金额的总量或日/月配额字段，并且统一作用于所选用户账户。
- 告警中心必须提供订阅配置与主动提醒能力；订阅规则至少需要支持渠道异常、API Key 异常、账户预警和失败率异常四类告警的启停配置，并提供统一的事件流或主动提醒入口。
- 当 `APP_ENV` 为 `prod` 或 `production` 时，`SESSION_SECRET_KEY` 与 `API_KEY_ENCRYPTION_SECRET` 必须使用非默认高强度随机值；若仍为空、仍为占位值或长度不足，应用必须拒绝启动。
- 对外正式支持的 OpenAI 兼容能力必须与实际 `/v1/*` 路由完全一致；未正式开放的能力禁止继续保留在对外能力声明、主要筛选项或正式流量统计口径中形成“半实现”状态。
- 渠道或模型若对外宣称支持 OpenAI 兼容文本、流式、`/v1/responses` 或图片能力，必须分别完成独立验收；禁止仅因文本对话可用，就默认判定流式、`/v1/responses` 或图片链路同样可用。
- 对外 `/v1/chat/completions` 与 `/v1/responses` 必须继续支持标准 SSE 流式返回，并在日志中保留 `is_stream`、首 Token 延迟和流式响应留痕能力，禁止为了简化实现而退回只支持一次性 JSON 返回。
- 对外代理路由必须遵守 provider 容量配置；若 provider 达到活跃请求、活跃流式请求、QPS 或失败率上限，路由候选必须排除该 provider，无法找到可用候选时必须返回明确限流或服务不可用错误。
- 对外代理请求获取 provider 容量租约后，必须在请求正常结束、上游异常、客户端中断或流式生成器退出时释放租约，禁止异常路径遗留活跃计数。
- 对外 `/v1/*` 实时并发计数、并发租约和短窗口 QPS/RPM 限流必须使用 Redis 等共享状态实现，禁止在 Gunicorn 多 worker 生产链路中仅依赖进程内存计数。
- 生产环境中会影响 `/v1/*` 路由、模型列表、provider 状态或权限判断的一致性缓存必须使用 Redis 等共享缓存；仅低风险、可短期不一致且不影响正式流量决策的派生数据允许退回进程内缓存。
- Redis 并发租约释放必须具备幂等性；释放前必须确认租约存在，并通过原子脚本或等价机制同时删除租约和递减相关计数，禁止重复释放导致计数为负。
- 对外 Bearer API Key 鉴权缓存禁止保存原始明文 Key；缓存 Key 必须基于 SHA256 哈希，并且 API Key、用户账户、授权 provider、模型白名单、额度或余额等关键写操作必须主动失效受影响缓存。
- SSE 流式请求必须使用独立流式并发租约，并在生成器 `finally` 中释放全局/API Key/账户/provider 容量租约；客户端取消必须记录 499 或等价状态且不得计为 provider 失败。
- SSE 流式请求必须支持连接上游超时、首 Token 超时、空闲超时和最大持续时间控制；流式 read timeout 可保持不限制，但必须在生成器层终止异常长连接，并在日志或 trace 中保留首 Token 延迟、总耗时和终态错误码。
- 生产可观测性必须以 Redis 全局活跃计数、数据库健康、连接池状态、后台任务积压、请求错误率和 provider 失败率为核心指标；`RuntimeStateService` 仅允许作为当前 worker 进程内参考，禁止作为多 worker 全局容量依据。
- 监控类异常必须写入统一 `alert_events` 告警事件流，并保留时间、对象、摘要、严重级别和处理状态；禁止另建第二套脱离告警中心的监控告警表。
- 思维等级字段必须统一收敛为 `无`、`low`、`medium`、`high`、`xhigh` 五档；其中 `无` 表示不指定，若请求参数显式传入思维等级，则日志与展示链路必须按原值透传记录。
- `使用文档` 页面必须优先面向外部接入用户，聚焦生产接入地址、鉴权方式、标准 `/v1/*` 路径、第三方客户端配置与一键复制能力；禁止继续以后台模块说明或帮助中心杂项为主体结构。

## 5. Git 与 GitHub 提交规范

- 需要将项目上传或同步到 GitHub 时，必须先检查本地仓库状态、当前分支、远程配置和提交人信息，再执行推送。
- 本地存在未提交改动时，必须先整理为明确提交，再进行远程绑定或推送，禁止带未提交工作区直接发布。
- 若项目尚未配置远程仓库，必须先新增 `origin` 后再执行推送。
- 本项目按单人维护模式处理 GitHub 同步：例行提交和推送时禁止主动执行 `git fetch`、`git pull`、`git merge`、`git rebase` 或 unrelated histories 合并流程。
- 需要执行标准上传流程时，必须按以下顺序操作：`git status -sb`、`git branch --show-current`、`git remote -v`、`git log --oneline -3`、`git add .`、`git commit -m "<清晰提交信息>"`、`git push -u origin main`。
- 若推送时出现 `fetch first`、`non-fast-forward`、`rejected` 或远程历史不一致提示，必须暂停并向用户说明远程状态异常；未获用户明确确认前，禁止自动合并、变基或强制推送。
- 仅在用户明确要求覆盖远程历史时，才允许使用 `--force-with-lease` 或等效受保护强制推送；禁止使用无保护的 `--force`。

## 6. Linux 部署与阿里云启动规范

- 面向 Ubuntu / 阿里云 ECS 的一键部署脚本必须放在项目根目录，优先使用 `start_aliyun.sh` 作为统一入口。
- Linux 部署脚本必须继续复用项目根目录下的 `.venv`、`.env`、`requirements.txt` 和 `data/`，禁止额外引入第二套独立运行目录。
- 一键部署脚本在项目根目录不存在 `.venv` 时，必须先创建 `.venv`，并使用国内镜像源安装依赖。
- 一键部署脚本必须默认将应用绑定到 `127.0.0.1:8000`，并通过 `nginx` 在 `80` 端口对外反向代理，禁止默认直接将 `uvicorn` 裸暴露到公网。
- 一键部署脚本必须写入并启用统一命名的 `systemd` 服务 `aotu-gpt.service`，保证后端开机自启和常驻运行。
- 生产环境 `aotu-gpt.service` 必须使用 `Gunicorn + UvicornWorker` 多 worker 启动，并默认通过 `WEB_CONCURRENCY`、`GUNICORN_TIMEOUT`、`GUNICORN_KEEPALIVE`、`GUNICORN_GRACEFUL_TIMEOUT` 控制 worker 数与超时参数；Windows `run.ps1` 仅允许作为本地开发启动方式。
- 生产环境 Web worker 启动阶段禁止执行 `create_all`、自动 DDL 或内置迁移逻辑；数据库结构变更必须通过独立迁移或初始化步骤完成后，再启动或滚动发布 Web worker。
- 一键部署脚本默认禁止自动执行数据库初始化或迁移；仅允许在显式设置 `RUN_DB_INIT=1` 时执行独立数据库初始化，且执行前必须先生成 PostgreSQL 备份。
- 生产环境 `aotu-gpt.service` 必须设置文件描述符限制，当前统一使用 `LimitNOFILE=1048576`。
- 一键部署脚本必须写入并启用统一命名的 `nginx` 站点配置 `aotu-gpt`，并在启用前移除默认站点，避免 `server_name _` 冲突。
- 一键部署脚本写入的 `nginx` 反向代理必须保留 SSE 透传配置：`proxy_buffering off`、`proxy_request_buffering off`、`proxy_read_timeout 600s`、`proxy_send_timeout 600s`。
- 若 `.env` 中 `LOCAL_PROXY_API_KEY` 为空，一键部署脚本允许自动生成随机值并写回 `.env`。
