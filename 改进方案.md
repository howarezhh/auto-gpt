# aotu-gpt 1000 并发完整改进方案

## 1. 改造目标

本方案用于指导 `aotu-gpt` 从当前单机中小并发代理系统，改造成具备同时处理 1000 个活跃请求能力的生产可用系统。这里的 `1000 并发` 指同一时刻最多 1000 个请求处于处理中，包括普通 JSON 请求、图片请求和 SSE 流式请求；不是 1000 RPS，也不是 1000 个用户在线。若平均请求耗时 30 秒，1000 并发约等于 33 RPS；若平均请求耗时 60 秒，1000 并发约等于 16 RPS。由于当前 provider 数量较少，实施阶段不做真实高并发压测，优先完成架构功能、限流能力、并发保护、后台化和可观测能力；真实 1000 并发只在 provider 容量充足后再做验收。LLM 网关的核心压力来自长耗时、长连接、上游不稳定、日志写入和计费后处理，因此本方案围绕“主链路轻量、数据库不锁、并发可控、流式隔离、日志计费后台化、可后续验收”展开。

改造后必须继续保持项目现有业务边界：对外统一使用 `/v1/*` OpenAI 兼容入口和外部 Bearer API Key 鉴权；管理后台与用户端继续使用服务端 Session 登录态；用户账户、API Key、模型目录、渠道授权、价格倍率、日志中心、账单、用户端使用文档和后台审计继续沿用现有功能口径。改造重点不是改变产品功能，而是改变运行架构和请求链路，让现有功能在 1000 并发下稳定运行。

## 2. 当前状态与瓶颈定位

当前项目使用 FastAPI、SQLAlchemy、SQLite、httpx AsyncClient 和 APScheduler。代理入口在 `app/routers/proxy.py`，核心转发逻辑在 `app/services/proxy_service.py`，API Key 鉴权与配额判断在 `app/services/api_key_service.py`，日志写入在 `app/services/log_service.py`，Token 回填在 `app/services/token_usage_service.py`，数据库连接在 `app/database.py`，运行配置在 `app/config.py`，Linux 部署入口在 `start_aliyun.sh`，Windows 启动入口在 `run.ps1`。这些文件就是本方案主要落点。

当前第一瓶颈是 SQLite。默认 `DATABASE_URL=sqlite:///./data/app.db`，虽然 `app/database.py` 对 SQLite 开启了 WAL，但 SQLite 仍然是单写者模型。1000 并发下，请求日志、provider 状态、API Key 用量、Token 回填、账单扣费都会争抢写锁，最容易出现排队、响应变慢、`database is locked` 或后台任务堆积。生产环境必须把 SQLite 降级为本地开发模式，正式并发目标必须使用 PostgreSQL。

当前第二瓶颈是单进程启动。`start_aliyun.sh` 的 systemd `ExecStart` 直接运行 `uvicorn app.main:app --host 127.0.0.1 --port 8000`，`run.ps1` 也是单进程启动。单进程无法充分利用多核 CPU，也无法通过 worker 隔离慢请求。1000 并发目标下，应改为 Gunicorn + UvicornWorker 多 worker，或在容器化后运行多个应用实例。

当前第三瓶颈是实时配额依赖数据库聚合。`ApiKeyService.get_daily_usage_snapshot` 和 `_validate_rate_limits` 会基于 `request_logs` 做请求数、Token、费用、QPS、RPM、TPM 判断。并发上升后，日志表既要写入又要被频繁聚合查询，会拖慢鉴权入口。实时 QPS/RPM/并发计数必须交给 Redis，数据库聚合只能作为后台对账和报表依据。

当前第四瓶颈是日志、Token 和计费在请求链路附近发生。`LogService.create_log` 会写入 `request_logs`，并触发 `TokenUsageService.enqueue_log_finalize`。这个方向已经有后台化基础，但 1000 并发下仍需要更明确地把 Token 统计、账单结算、余额扣减、日志派生字段刷新从用户请求主链路中剥离出来。主链路应只做必要日志和事件投递，重计算放后台任务。

当前第五瓶颈是流式请求没有独立容量池。`ProxyService.forward_stream_request` 会长时间保持客户端连接和上游连接，流式请求越多，占用 worker、socket、上游连接池和内存越久。1000 并发不能让流式请求无限制占满全部容量，必须设置全局、账户、API Key、provider 维度的流式并发上限。

## 3. 目标架构

1000 并发目标不需要直接引入 Kubernetes、Kafka、ClickHouse 和多区域架构，但必须升级为生产级单集群结构。推荐基础架构是：`Nginx -> Gunicorn/Uvicorn 多 worker -> FastAPI -> PostgreSQL -> Redis -> 后台任务`。Nginx 负责公网入口、反向代理、连接超时、请求体限制和 SSE 透传；FastAPI 负责 API Key 鉴权、路由选择、上游转发和基础日志；PostgreSQL 存储用户、API Key、模型、渠道、配置、账单和请求日志；Redis 负责并发计数、QPS/RPM 限流、短期缓存和租约；后台任务负责 Token 回填、账单结算、余额同步、日志补全、健康检查和保留期清理。

推荐部署拓扑分两档。最低可落地拓扑是单台 8 核 16G 服务器运行应用、Nginx、Redis 和 PostgreSQL，但这只是成本优先方案，适合验证 1000 并发设计。更稳妥的生产拓扑是应用服务器、PostgreSQL 服务器和 Redis 服务器分开部署：应用服务器 8 核 16G，PostgreSQL 4 核 16G 或更高并使用 SSD，Redis 2 核 4G 或更高。若后续流式请求长期超过 300 并发，应把应用服务拆成普通请求实例和流式请求实例。

## 4. 实施总顺序

实施必须按“先功能自检，再替换，再限流，再后台化，再小流量验证”的顺序推进。第一阶段建立当前功能基线，不做真实高并发压测，只确认现有关键接口、provider、API Key、日志和计费链路可用。第二阶段迁移 PostgreSQL 和连接池，解决数据库写锁风险。第三阶段改为 Gunicorn 多 worker，解决单进程瓶颈。第四阶段引入 Redis 并发计数和 QPS/RPM 限流，解决实时入口压力。第五阶段改造日志、Token 和计费后台化，降低请求结束时的数据库写入和计算压力。第六阶段专项治理流式请求，确保流式请求受到并发上限保护，不会无限制占满普通请求容量。第七阶段完成监控、回滚和生产参数固化。真实 1000 并发验收不在当前 provider 容量不足时执行。

每个阶段都必须单独提交、单独做功能自检、单独保留回滚方案。不要一次性把数据库、Redis、队列、多 worker、流式限流全部改完，否则出现功能退化时无法定位原因。每阶段功能验收通过后再进入下一阶段；真实压力验收必须等 provider 容量充足后单独安排。

## 5. 阶段 0：功能基线

阶段目标是建立当前系统功能基线，不做真实高并发压测。先用现有 `.venv` 启动项目，在测试环境准备一个可用 API Key、一个可用 provider、一个目标模型，分别验证 `/v1/models`、`/v1/chat/completions` 非流式、`/v1/chat/completions` 流式、`/v1/responses` 非流式和 `/v1/responses` 流式。每类接口只做少量请求验证，确认认证、路由、上游调用、日志记录、Token 回填和账单链路没有明显错误。

建议新增轻量自检脚本 `scripts/smoke_check.py` 或 `tools/smoke_check.py`，使用 `httpx.AsyncClient` 串行或小并发验证接口可用性。脚本参数应包括：`--base-url`、`--api-key`、`--endpoint`、`--model`、`--stream`、`--timeout-sec`。脚本输出必须包含请求是否成功、状态码、trace_id、响应耗时和错误样例。脚本不能写死真实 API Key，必须从命令行参数或环境变量读取。若后续 provider 容量允许，可再单独新增 `scripts/load_test.py` 做真实压力验收，但当前阶段不执行。

阶段验收标准是形成一份可复现功能基线：现有 SQLite + 单进程 Uvicorn 下，关键接口均能完成一次或少量调用，日志、Token、账单和后台状态可追踪。若少量请求已经出现数据库锁、上游错误或日志异常，就必须先修复基础功能再继续并发架构改造。

## 6. 阶段 1：PostgreSQL 迁移

阶段目标是把生产数据库从 SQLite 迁移到 PostgreSQL，同时保留 SQLite 本地开发能力。需要修改 `requirements.txt`，增加 `psycopg[binary]`；修改 `app/config.py`，新增数据库连接池配置项：`db_pool_size`、`db_max_overflow`、`db_pool_timeout`、`db_pool_recycle`；修改 `app/database.py`，让 SQLite 继续使用 `check_same_thread=False` 和 WAL，非 SQLite 则使用 SQLAlchemy 连接池参数。生产环境必须在 `APP_ENV=prod` 或 `production` 时拒绝默认 SQLite 启动，避免误把正式流量打到本地文件数据库。

本次阶段 1 已按用户指定密码落地默认配置：`DATABASE_URL=postgresql+psycopg://aotu_gpt:zhh123456@127.0.0.1:5432/aotu_gpt`、`DB_POOL_SIZE=30`、`DB_MAX_OVERFLOW=70`、`DB_POOL_TIMEOUT=5`、`DB_POOL_RECYCLE=1800`。PostgreSQL 侧使用独立数据库 `aotu_gpt` 和独立用户 `aotu_gpt`；本地开发或 Windows 验证可使用 `docker-compose.postgres.yml` 启动 PostgreSQL 16，阿里云部署由 `start_aliyun.sh` 自动安装 PostgreSQL、创建用户、创建数据库并写入同一套连接配置。若 PostgreSQL 与应用不在同机，必须确认内网延迟稳定且安全组只允许应用服务器访问数据库端口。

迁移策略当前采用“PostgreSQL 空库初始化 + 配置重建”。项目暂未引入正式 Alembic 迁移体系，`init_database()` 会先执行 `Base.metadata.create_all` 创建 PostgreSQL 表结构，再初始化系统设置、同步 provider 模型和模型目录；历史 SQLite 字段补齐逻辑只允许在 SQLite 会话下执行，禁止把 `PRAGMA` 和 SQLite 专用 `ALTER TABLE` 跑到 PostgreSQL。若后续必须迁移历史 SQLite 数据，应单独写迁移脚本：从 SQLite 读取表数据，按模型字段插入 PostgreSQL，遇到自增 ID、外键、时间字段和 Decimal 字段时逐表处理。迁移前必须备份 `data/app.db`、`data/app.db-wal`、`data/app.db-shm`。

阶段 1 当前实现清单：`requirements.txt` 已增加 `psycopg[binary]`；`app/config.py` 已新增数据库连接池配置，并在生产环境拒绝 SQLite；`app/database.py` 已按 SQLite 与 PostgreSQL 分支创建 SQLAlchemy engine；`app/main.py` 已避免 PostgreSQL 执行 SQLite 专用迁移，并将共享钱包回填改为跨数据库表结构检查；`.env` 与 `.env.example` 已切换为 PostgreSQL 默认连接；`docker-compose.postgres.yml` 已提供本地 PostgreSQL 16；`start_aliyun.sh` 已补齐 PostgreSQL 安装、启动、用户创建、数据库创建和 `.env` 写入。

阶段验收标准是应用使用 PostgreSQL 启动成功，管理后台可登录，API Key 可创建，provider 和模型可配置，`/v1/models`、非流式和流式代理均可用。当前阶段不做并发压测，只做小流量功能验证，并确认 PostgreSQL 连接数、日志写入、账单记录和后台任务行为正常。真实连接池承载能力留到 provider 容量充足后再验收。

## 7. 阶段 2：多 worker 启动与部署脚本改造

阶段目标是把生产启动从单进程 Uvicorn 改为多 worker。需要修改 `requirements.txt` 增加 `gunicorn`；修改 `start_aliyun.sh` 的 systemd `ExecStart`，从 `python -m uvicorn app.main:app --host 127.0.0.1 --port 8000` 改为 `python -m gunicorn app.main:app -k uvicorn.workers.UvicornWorker -w ${WEB_CONCURRENCY:-4} -b 127.0.0.1:8000 --timeout ${GUNICORN_TIMEOUT:-120} --keep-alive ${GUNICORN_KEEPALIVE:-75}`。Windows `run.ps1` 可继续用于本地开发，不要求改成 Gunicorn，但必须在文档中明确 Windows 脚本不是生产并发启动方式。

建议新增配置项：`WEB_CONCURRENCY=4`、`GUNICORN_TIMEOUT=120`、`GUNICORN_KEEPALIVE=75`、`GUNICORN_GRACEFUL_TIMEOUT=30`。4 核机器从 2 到 4 worker 开始，8 核机器从 4 worker 开始。worker 数不是越大越好，过多 worker 会增加数据库连接、内存和上下文切换。上游请求耗时长时，多 worker 可以提高并发承载，但必须配合数据库连接池和上游连接池。

Nginx 保持反代到 `127.0.0.1:8000`，但必须确认 SSE 设置不被破坏：`proxy_buffering off`、`proxy_request_buffering off`、`proxy_read_timeout 600s`、`proxy_send_timeout 600s`、`client_max_body_size 20m`。Linux 服务还需要设置文件描述符限制，systemd 服务建议加入 `LimitNOFILE=1048576`。如果只做 1000 并发，`ulimit -n >= 65535` 通常够用，但为了流式和上游连接冗余，生产建议直接提高。

阶段验收标准是 systemd 启动后有多个 worker 进程，`/live`、`/ready`、`/health` 正常，Nginx 访问正常，SSE 流式不被缓冲。当前阶段只做少量请求功能验证，确认多 worker 下 Session、静态资源、后台任务、数据库连接和上游代理均正常。CPU、内存和连接数的压力指标留到后续容量验收时确认。

阶段 2 当前实现清单：`requirements.txt` 已增加 `gunicorn`；`.env` 与 `.env.example` 已增加 `WEB_CONCURRENCY=4`、`GUNICORN_TIMEOUT=120`、`GUNICORN_KEEPALIVE=75`、`GUNICORN_GRACEFUL_TIMEOUT=30`；`app/config.py` 已显式声明这些配置项；`start_aliyun.sh` 写入的 `aotu-gpt.service` 已从单进程 Uvicorn 切换为 `Gunicorn + UvicornWorker` 多 worker 启动，并通过 `/bin/bash -lc` 支持 `${WEB_CONCURRENCY:-4}` 等默认值展开；systemd 已加入 `LimitNOFILE=1048576`；Nginx 已加入 `proxy_request_buffering off` 并保留 SSE 所需超时与缓冲配置；`README.md` 与 `启动指南.md` 已明确 Windows `run.ps1` 仅用于本地开发，不作为生产并发启动方式。

## 8. 阶段 3：上游连接池与 provider 容量

阶段目标是让上游连接池覆盖 1000 并发，并防止单个 provider 被打爆。需要修改 `app/config.py`，把当前 `upstream_max_connections`、`upstream_max_keepalive_connections`、`upstream_pool_timeout_s` 做成生产可调配置，并在 `.env.example` 中给出 1000 并发建议值。推荐初始配置是：`UPSTREAM_MAX_CONNECTIONS=1200`、`UPSTREAM_MAX_KEEPALIVE_CONNECTIONS=300`、`UPSTREAM_POOL_TIMEOUT_S=10`、`REQUEST_TIMEOUT_MS=60000`。如果流式请求多，`REQUEST_TIMEOUT_MS` 对非流式有效，流式读超时仍需单独控制。

需要在 provider 或 provider_model 层增加运行时容量控制字段或服务层配置：`max_active_requests`、`max_active_streams`、`max_qps`、`max_error_rate`、`first_token_timeout_sec`。如果短期不改数据库表，可先用系统设置或内存默认值实现，但生产上应落到 provider 配置中，便于后台管理。路由服务选择候选时必须排除已超过活跃并发上限的 provider；请求结束后必须释放 provider 活跃计数。

阶段验收标准是 provider 级容量配置可以保存、读取并参与路由判断；当某个 provider 达到配置上限时，请求会自动路由到其他 provider 或返回 429/503，不会继续无限制打向同一上游。当前阶段可以通过低阈值人工验证，例如把某 provider 上限设为 1，用少量请求确认限流逻辑生效，不需要真实高并发压测。

阶段 3 当前实现清单：`app/config.py`、`.env`、`.env.example` 和 `start_aliyun.sh` 已补齐 `REQUEST_TIMEOUT_MS=60000`、`UPSTREAM_MAX_CONNECTIONS=1200`、`UPSTREAM_MAX_KEEPALIVE_CONNECTIONS=300`、`UPSTREAM_POOL_TIMEOUT_S=10`；`providers` 已新增 `max_active_requests`、`max_active_streams`、`max_qps`、`max_error_rate`、`first_token_timeout_sec` 字段，并通过启动迁移补齐旧库字段；后台中转站表单和列表已支持保存、回显和展示容量配置与运行时活跃计数；`RouterService` 会在候选排序后排除超过活跃请求、流式请求、QPS 或失败率上限的 provider；`ProxyService` 会在 JSON 与 SSE 转发期间获取 provider 容量租约，并在请求结束、异常或客户端中断时释放计数；SSE 首 Token 已按 provider 的 `first_token_timeout_sec` 做运行时控制。

当前阶段的 provider 容量计数使用进程内存实现，适合低阈值功能验证；在 Gunicorn 多 worker 生产模式下，它只能约束单个 worker 内的活跃计数，不能作为全局精确容量控制。阶段 4 引入 Redis 后，应把 provider 活跃请求、活跃流式和 QPS 计数迁移为跨 worker 原子计数。

## 9. 阶段 4：Redis 并发计数与限流

阶段目标是把实时并发和短窗口限流从数据库迁移到 Redis。需要新增依赖 `redis` 或 `redis[hiredis]`；新增 `app/services/redis_service.py` 管理 Redis 客户端生命周期；新增 `app/services/concurrency_service.py` 管理活跃请求租约；新增 `app/services/rate_limit_service.py` 管理 QPS/RPM 限流。`app/main.py` 的 lifespan 需要初始化和关闭 Redis 客户端，健康检查 `/ready` 需要增加 Redis 可用性检查。

并发租约设计必须可恢复。请求进入时生成 `request_id` 或复用现有 request_id，写入 `concurrency:lease:{request_id}`，TTL 建议设置为请求最大允许时长加冗余，例如 900 秒；同时递增 `concurrency:global:active`、`concurrency:api_key:{id}:active`、`concurrency:account:{id}:active`、`concurrency:provider:{id}:active`。请求结束、异常、客户端断开时删除租约并递减计数。为避免重复释放，释放逻辑必须检查租约存在并用 Lua 脚本原子处理。

限流 Key 建议为：`rate:qps:{api_key_id}:{unix_second}`、`rate:rpm:{api_key_id}:{yyyyMMddHHmm}`、`rate:provider:qps:{provider_id}:{unix_second}`。QPS/RPM 判断必须用 Redis 原子 `INCR + EXPIRE` 或 Lua 脚本实现。超限返回 OpenAI 兼容错误，HTTP 状态码 429，错误码建议为 `rate_limit_exceeded`、`concurrency_limit_exceeded` 或 `stream_concurrency_exceeded`。错误响应仍必须带 `X-Trace-Id`，并按现有日志规范记录认证或限流失败。

推荐初始配置为：`GLOBAL_MAX_ACTIVE_REQUESTS=1000`、`GLOBAL_MAX_ACTIVE_STREAMS=300`、`API_KEY_MAX_ACTIVE_REQUESTS=50`、`API_KEY_MAX_ACTIVE_STREAMS=10`、`ACCOUNT_MAX_ACTIVE_REQUESTS=100`、`ACCOUNT_MAX_ACTIVE_STREAMS=20`、`PROVIDER_MAX_ACTIVE_REQUESTS=300`、`PROVIDER_MAX_ACTIVE_STREAMS=150`。这些值必须通过 `.env` 和系统设置可调，不能硬编码。

阶段验收标准是 Redis 并发计数、租约、释放和超限返回逻辑可通过小流量验证。可以把 `GLOBAL_MAX_ACTIVE_REQUESTS` 临时设为 1 或 2，发起少量并行请求，确认超限请求稳定返回 429，正常请求不受影响，请求结束后 Redis 活跃计数回到 0 或接近 0。真实 1000 并发只作为未来 provider 容量充足后的验收项目。

阶段 4 当前实现清单：`requirements.txt` 已增加 `redis[hiredis]`；`.env`、`.env.example`、`app/config.py` 和 `start_aliyun.sh` 已补齐 `REDIS_URL`、`CONCURRENCY_LEASE_TTL_SECONDS`、全局/API Key/账户/provider 活跃请求与流式请求上限；`docker-compose.postgres.yml` 已增加本地 Redis 服务；`app/services/redis_service.py` 管理 Redis 客户端生命周期；`app/services/concurrency_service.py` 使用 Redis Lua 脚本原子获取和释放并发租约；`app/services/rate_limit_service.py` 使用 Redis `INCR + EXPIRE` 实现 API Key QPS/RPM 与 provider QPS 短窗口限流；`/v1/chat/completions`、`/v1/responses` 和 `/v1/models` 已接入全局、API Key、账户级并发租约，SSE 流式响应会在生成器退出时释放租约；provider 容量服务已优先读取 Redis 中的 provider 活跃请求、流式请求和 QPS 计数；`/ready` 已加入 Redis 可用性和 Redis 全局活跃计数信息。

阶段 4 当前只完成代码级与配置级验证，不执行真实并发压测。本地若未启动 Redis，`/ready` 会显示 Redis 不可用；外部 `/v1/*` 在需要 Redis 并发或 QPS/RPM 判断时会返回明确的 Redis 不可用错误。生产部署必须先确认 Redis 正常，再逐步把阈值调低做少量人工验证。

## 10. 阶段 5：API Key 鉴权缓存

阶段目标是减少每个请求进入时对数据库的重复读取。需要在 `ApiKeyService.authenticate_request` 前后引入缓存层：根据 Bearer Key 的 SHA256 哈希读取 `auth:key_hash:{hash}`，缓存命中时返回包含 API Key ID、启停状态、owner_user_id、授权 provider、允许模型、允许 endpoint、路由上下文、QPS/RPM/TPM、账户启停状态和账户额度摘要的认证上下文。缓存 TTL 建议 30 到 120 秒。

缓存失效必须覆盖所有关键写操作。API Key 新增、编辑、启停、轮换、删除、批量授权更新后必须清理对应缓存；用户账户禁用、额度调整、余额调整后必须清理该用户下所有 API Key 缓存；provider 授权变化、模型白名单变化、策略模板套用后必须清理受影响缓存。短期可以在相关服务方法中调用 `ApiKeyAuthCache.invalidate_api_key(id)` 和 `invalidate_user(user_id)`；长期可通过审计事件或统一领域事件清理。

鉴权缓存不能绕过安全规则。原始 Key 仍然必须先做 Bearer 格式解析和 SHA256 哈希；缓存中不得保存原始明文 Key；可回显明文仍按项目现有加密存储规则处理。对余额、总额度等强一致要求高的字段，缓存只能作为快速快照，实时消耗仍要结合 Redis 热计数和后台对账。

阶段验收标准是少量请求下 API Key 鉴权缓存可以命中，数据库重复查询减少；修改 API Key 状态后缓存能在 TTL 内或主动失效后生效；禁用账户或 Key 后不会继续放行请求。当前不通过高并发压测证明缓存收益，只通过日志、调试计数或临时指标确认缓存逻辑正确。

阶段 5 当前实现清单：`app/config.py`、`.env` 和 `.env.example` 已增加 `API_KEY_AUTH_CACHE_TTL_SECONDS=60`；`app/services/api_key_auth_cache.py` 使用 Redis 按 `auth:key_hash:{sha256}` 缓存鉴权上下文，并维护 API Key ID 与用户 ID 到缓存 Key 的失效索引；`ApiKeyService.authenticate_request` 保留 Bearer 解析和 SHA256 哈希校验，缓存命中时重建 API Key 鉴权上下文，缓存未命中时沿用数据库鉴权并写入短 TTL 快照；缓存内容不保存原始明文 Key。API Key 新增后的首次请求会建立缓存，编辑、启停、轮换、删除、批量启停、批量删除、批量渠道授权更新、策略模板套用、过期处理、余额调整，以及用户启停、删除、账户额度和余额调整都会主动清理对应 API Key 或用户下全部 API Key 缓存。

阶段 5 当前只完成代码级验证，不执行真实高并发压测。本地若未启动 Redis，鉴权缓存会自动旁路回数据库鉴权，但阶段 4 的 Redis 并发与 QPS/RPM 限流仍会在正式 `/v1/*` 请求链路返回明确的 Redis 不可用错误；生产部署必须先确认 Redis 正常，再通过禁用 API Key、禁用账户和低频重复请求验证缓存命中与主动失效。

## 11. 阶段 6：日志、Token 和计费后台化

阶段目标是降低主链路数据库写入和计算压力。短期建议保留 `request_logs` 同步插入最小字段，避免日志完全丢失；同时把 Token 回填、费用计算、余额扣减、日志派生字段刷新放入后台任务。现有 `TokenUsageService.enqueue_log_finalize` 已经提供入口，应扩展为更稳定的队列。若不立即引入 Celery，可先用 Redis Queue 或 APScheduler + Redis 去重；中期推荐 Celery/RQ/Arq。

日志最小字段应包括：`trace_id`、`request_id`、`api_client_key_id`、`user_account_id`、`provider_id`、`requested_model`、`request_path`、`http_method`、`is_stream`、`has_image`、`success`、`status_code`、`latency_ms`、`ttfb_ms`、`duration_ms`、`attempt_count`、`error_type`、`error_code`、`created_at`。大字段 `request_body_json`、`response_body_json`、`response_text` 必须受 `enable_payload_logging`、`enable_stream_response_persist` 和 `max_logged_body_bytes` 控制，高并发生产默认建议关闭完整 payload 或只保存失败请求。

计费后台任务必须幂等。建议以 `request_log_id` 或 `request_id` 建唯一处理标记，新增字段或单独表记录 `billing_finalized_at`、`billing_event_id`、`billing_attempt_count`。如果同一个日志被重复处理，不能重复扣减账户余额。余额扣减建议在同一个数据库事务中完成：读取日志、计算费用、写账单记录、更新 API Key 总费用和用户余额、标记日志计费状态。失败任务进入重试，超过重试次数写入告警。

Token 回填任务应优先使用上游返回的 usage；上游未返回 usage 时再用 `tiktoken` 估算。高并发时不应在请求主链路做大文本 Token 计算。流式响应可以只保留截断后的文本片段用于估算，或在无法准确估算时标记 `billing_status=pending_tokens`，由后台补偿任务处理。

阶段验收标准是少量真实请求下，日志、Token 回填、计费、余额扣减和账单记录均能由后台任务正确完成；重复执行计费任务不会重复扣费；失败任务有重试和告警；日志中心仍能查到完整核心字段。当前不通过 1000 并发压测证明主链路收益，但必须确保后台化不会破坏业务正确性。

阶段 6 当前实现清单：`request_logs` 已增加 `billing_finalized_at`、`billing_event_id`、`billing_attempt_count`、`billing_error`、`token_finalize_attempt_count`、`token_finalize_error`，用于记录后台 Token/计费任务状态；`TokenUsageService.enqueue_log_finalize` 继续作为统一入口，并增加 Redis 去重 Key 与 APScheduler 延迟任务，失败后按短退避重试，超过最大次数写入告警事件；`BillingService.finalize_request_log_billing` 以 `request_log_id` 查找既有账单并按金额 delta 更新 API Key 累计消费和用户共享余额，重复执行不会重复扣费；日志主链路保留 `request_logs` 最小核心字段，同步写入后由后台任务补齐 Token、派生字段、费用和账单。

阶段 6 当前只做代码级验证和幂等小样例验证，不执行真实 1000 并发压测。生产建议保持 `enable_payload_logging=false`、`enable_stream_response_persist=false`，只在排障时短期开启完整 payload 或流式文本持久化，并通过 `max_logged_body_bytes` 控制最大记录体积。

## 12. 阶段 7：流式请求专项治理

阶段目标是确保 SSE 流式请求不会占满所有资源。建议初始容量分配为总活跃请求 1000，其中流式请求上限 200 到 300，普通非流式请求保留 700 到 800。`ProxyService.forward_stream_request` 进入前必须获取流式租约，成功建立上游流后持续转发，结束、异常、取消都必须释放租约。客户端断开会触发 `asyncio.CancelledError`，释放逻辑必须放在 `finally`。

流式超时建议分三类：连接上游超时、首 Token 超时、空闲超时。建议配置为 `STREAM_CONNECT_TIMEOUT_SECONDS=10`、`STREAM_FIRST_TOKEN_TIMEOUT_SECONDS=60`、`STREAM_IDLE_TIMEOUT_SECONDS=120`、`STREAM_MAX_DURATION_SECONDS=600`。现有 `_build_httpx_timeout` 对流式 read timeout 设置为 None，后续应在流式生成器层实现空闲超时和最大持续时间，不要让异常长连接永久占用资源。

流式日志策略必须轻量。禁止逐 chunk 写数据库；只在流式结束时写一条摘要日志，记录首 Token 延迟、总耗时、状态、错误码、Token、截断响应文本和 trace。若客户端取消，应记录 499 或等价客户端取消状态，不应计为 provider 失败；若上游中断，应记录 502 或 `stream_interrupted`，并按 provider 失败处理。

阶段验收标准是少量流式请求下，流式租约获取、释放、客户端取消、上游异常和日志记录都正确；把 `GLOBAL_MAX_ACTIVE_STREAMS` 临时设为 1 时，第二个流式请求能稳定返回 429；客户端主动断开后 Redis 活跃计数回落；流式日志完整记录首 Token 延迟和总时长。真实 200 到 300 个流式连接不在当前 provider 容量不足时执行。

阶段 7 当前实现清单：`.env`、`.env.example`、`app/config.py`、`start_aliyun.sh` 和系统设置模型已补齐 `STREAM_CONNECT_TIMEOUT_SECONDS=10`、`STREAM_FIRST_TOKEN_TIMEOUT_SECONDS=60`、`STREAM_IDLE_TIMEOUT_SECONDS=120`、`STREAM_MAX_DURATION_SECONDS=600`；`ProxyService.forward_stream_request` 在进入流式转发前已有 Redis 流式并发租约，`StreamingResponse` 生成器退出、异常或客户端取消后会在 `finally` 释放；provider 容量租约同样在流式生成器 `finally` 释放；流式上游连接使用独立 connect timeout，read timeout 保持 `None`，首 Token、空闲 chunk 与最大持续时间由生成器层 `asyncio.wait_for` 控制；客户端取消记录 499 和 `client_cancelled`，不计 provider 失败，首 Token、空闲或最大持续时间超时按上游流中断记录 502 并计 provider 失败；流式日志仍只在结束时写入一条摘要，不逐 chunk 写库。

阶段 7 当前只做代码级、配置级和低阈值验证准备，不执行真实 200 到 300 流式连接压测。生产验证时建议先把 `GLOBAL_MAX_ACTIVE_STREAMS` 调为 1，发起两个少量流式请求确认第二个返回 429，再验证客户端主动断开后 Redis 活跃流式计数回落。

## 13. 阶段 8：监控、健康检查和告警

阶段目标是让系统在 1000 并发下可观测、可定位、可告警。现有 `/live`、`/ready`、`/health` 和 `X-Active-Requests` 应继续保留，并扩展为 Redis 全局活跃请求、全局流式活跃请求、provider 活跃请求、Redis 状态、PostgreSQL 状态、后台任务积压状态。`RuntimeStateService` 的单进程计数只能表示当前 worker 内存状态，不能作为多 worker 全局并发依据。

建议新增 `/metrics` 或管理端内部指标接口，输出：请求总数、成功数、失败数、429 数、5xx 数、P50/P95/P99、活跃请求、活跃流式、provider 成功率、provider 失败率、平均首 Token 延迟、Redis 命令耗时、PostgreSQL 连接池使用率、后台任务队列长度、计费失败数、Token 回填失败数。生产上推荐接 Prometheus + Grafana；如果暂不引入，也要在管理端健康页展示核心指标。

告警规则建议：全局活跃请求超过 900 持续 1 分钟；流式活跃请求超过 300；5xx 错误率超过 1%；429 突增；provider 失败率超过 20%；Redis 不可用；PostgreSQL 连接池等待超时；后台任务积压超过 1000；计费失败超过 10；日志写入失败；余额对账异常。告警应写入现有告警中心，并保留时间、对象、摘要、严重级别和处理状态。

阶段验收标准是功能验证过程中能实时看到活跃请求、错误率和关键状态；故意关闭 Redis 或 PostgreSQL 时 `/ready` 能反映异常；后台任务积压能触发告警；少量请求结束后活跃请求指标恢复正常。大流量趋势指标留待后续真实容量验收。

阶段 8 当前实现清单：新增 `SystemMetricsService` 统一采集数据库可用性、数据库连接池状态、Redis 可用性与命令耗时、Redis 全局活跃请求、Redis 全局流式请求、单 worker 运行时计数、请求总数、成功数、失败数、429、5xx、P50/P95/P99、平均首 Token 延迟、provider 活跃请求/流式/QPS 与失败率、APScheduler 任务状态、Token/计费待补全积压和失败数；`/ready` 已改为基于系统指标返回 `ready` 或 `degraded`，并在 Redis 或数据库不可用时返回 503；新增公开 `/metrics` JSON 快照和后台 `/api/metrics/system`；后台“告警中心”已展示核心健康指标，并把 Redis 不可用、数据库不可用、5xx/429 异常、provider 失败率过高、后台任务积压、Token/计费失败写入 `alert_events`。

阶段 8 当前只做功能级与接口级验证，不执行真实上游中转站压测。后续生产验证可通过临时关闭 Redis/PostgreSQL、人工制造后台任务积压或插入少量失败日志确认 `/ready` 与告警中心状态变化。

## 16. 数据一致性与回滚策略

PostgreSQL 迁移前必须备份 SQLite 文件和 `.env`。如果迁移失败，应能把 `DATABASE_URL` 切回 SQLite 并用旧启动方式恢复服务。多 worker 上线前必须保留单进程 Uvicorn 的 systemd 命令或脚本参数，出现异常时能回滚。Redis 限流上线必须有开关，例如 `ENABLE_REDIS_RATE_LIMIT=false` 和 `ENABLE_DISTRIBUTED_CONCURRENCY=false`，如果 Redis 故障导致大量误拒绝，可以临时关闭分布式限流并降级到数据库/本地模式，但降级期间必须降低对外流量。

后台计费上线前必须先灰度。可以先只投递任务但不扣费，对比后台计算结果与原同步计算结果；确认一致后再启用后台扣费。后台扣费必须幂等，必须能重跑，必须能根据 `request_log_id` 查到处理状态。流式限流上线前必须先设较高阈值观察，再逐步调整到目标阈值，避免误伤正常用户。

## 17. 功能验证与后续容量验收方案

当前阶段不进行真实压测，因为 provider 数量和上游容量不足。功能验证必须覆盖四类流量：`/v1/models` 轻请求、非流式 chat、流式 chat、混合小流量请求。每类接口只做少量请求，目标是确认功能正确、限流可控、日志可追踪、计费不重复、异常可恢复。若项目实际图片请求占比高，需要做少量图片请求功能验证，因为图片请求体更大、上游耗时更长。

功能验证建议采用低阈值模拟，而不是真实高并发。例如把全局并发上限临时设为 1 或 2，验证并发超限返回 429；把流式并发上限临时设为 1，验证第二个流式请求被拒绝；把 provider 并发上限临时设为 1，验证路由转移或拒绝；重复执行同一个计费任务，验证不会重复扣费；临时关闭 Redis 或 PostgreSQL，验证 `/ready` 和告警能反映异常。

后续 provider 容量充足后，再单独安排容量验收。容量验收阶梯可以是 100、300、500、800、1000，每档先跑 10 分钟观察错误，再跑 30 分钟稳定性测试。最终验收要求 1000 活跃请求下代理自身 5xx 错误率小于 0.1%，429 只出现在主动限流场景，PostgreSQL 无连接池耗尽，Redis 无明显命令超时，后台任务积压可在请求结束后逐步消化，活跃请求计数回到 0 或接近 0。该容量验收不是当前实施阶段的动作。

## 18. 最终上线标准

当前功能上线前必须满足：生产环境使用 PostgreSQL，不使用默认 SQLite；生产启动使用 Gunicorn/Uvicorn 多 worker；Redis 限流和并发计数开启；上游连接池配置已按 1000 并发目标预留；流式请求有独立并发上限；日志、Token 和计费不在主链路执行重计算；`/ready` 能检查数据库和 Redis；管理端能看到核心健康指标；回滚方案经过至少一次演练。由于当前不做真实压测，不能对外宣称已经通过 1000 并发实测，只能说明功能和架构已按 1000 并发目标实现。

上线后第一周建议保守配置：全局并发按实际 provider 承载能力设置，流式并发从较低值开始，例如全局并发 100 到 300、流式并发 20 到 50，观察稳定后再逐步提高。若出现数据库连接池等待、Redis 命令超时、后台任务积压或 provider 429 激增，应先降全局并发和流式并发，再排查瓶颈。不要在没有 provider 容量和压测报告的情况下直接把阈值提高到 1000。

## 19. 结论

按“同时支持 1000 个请求访问”的目标，本项目不需要 10 万并发平台级架构，但必须完成 PostgreSQL、多 worker、Redis 限流和并发计数、后台任务、流式治理、监控告警和压测验收。最关键的逻辑是：数据库负责持久状态，Redis 负责实时计数，FastAPI 主链路只做鉴权、限流、路由和代理，日志/Token/计费交给后台任务，流式请求单独限制容量。

完成本方案各功能阶段后，项目可以说明“已按 1000 并发目标完成架构功能改造”，但不能说明“已经实测支持 1000 并发”。未完成 PostgreSQL、Redis 和多 worker 前，不应对外承诺 1000 并发；未完成真实容量验收前，只能说明架构设计和功能实现支持该目标，不能说明生产环境已经稳定承载。
