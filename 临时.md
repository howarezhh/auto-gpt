该系统目前已经从一个基础 FastAPI 代理雏形，升级成了一个面向 **GPT / OpenAI-compatible 中转站** 的统一代理调度平台。  
它现在能够完成的功能可以概括为下面几类。

**1. 统一对外代理能力**
系统对外提供统一的 `base_url + api_key`，用户可以像调用标准 OpenAI-compatible 接口一样发请求，而不需要关心背后具体命中了哪个中转站。

当前已支持的核心对外接口：

- `POST /v1/chat/completions`
- `GET /v1/models`
- `POST /v1/responses`
- `POST /v1/embeddings`
- `POST /v1/{subpath:path}` 作为兼容兜底

这意味着它已经可以承担“统一接入层”的角色。

**2. 文本、图片、多种请求透传**
系统已经实现了“透传优先”：

- 文本对话请求可直接转发
- 图片输入可按 OpenAI-compatible 多模态 `messages` 结构透传
- 不主动改写用户 payload
- 上游返回 JSON 就返回 JSON
- 上游返回 SSE 就原样转发 SSE

也就是说，对用户而言，调用体验已经尽量接近直接和 GPT 中转接口对话。

**3. 流式传输支持**
系统已经支持流式请求，并对流式链路做了稳定性增强。

当前流式能力包括：

- 支持 `stream: true`
- 首个上游建连失败时，可切换下一个 provider
- 流真正开始输出后，不再中途切换，避免内容错乱
- 原样透传 SSE 数据，不伪造 chunk
- 流式生命周期有状态追踪：
  - `connecting`
  - `stream_opened`
  - `first_chunk_received`
  - `finished`
  - `interrupted`
- 已记录首包耗时和总耗时
- 客户端中断时会关闭上游流，避免连接泄漏

**4. 中转站统一管理**
系统已经能管理多个 GPT 中转站 provider，包括：

- 新增、编辑、删除中转站
- 配置 `base_url`
- 配置 `api_key`
- 设置 provider 类型
- 设置 provider 优先级
- 设置 provider 权重
- 设置 timeout
- 设置 max retries
- 启用 / 禁用 provider
- 设置默认 provider
- 查看 provider 健康状态
- 查看 provider 熔断状态

同时，已经去掉了默认自动注入的假数据，不会再污染真实部署环境。

**5. 模型级管理**
系统不再只管理 provider，而是已经具备模型级资源管理能力。

每个 provider 下可以挂多个模型，并且每个模型都能独立配置：

- 模型名
- 是否启用
- 模型优先级
- 模型权重
- 是否支持 stream
- 是否支持 vision
- 模型健康状态
- 模型熔断状态
- 最近延迟
- 最近错误
- 最近检测时间

并且已经支持：

- 单独启停某个模型
- 单独调整某个模型权重
- 单独调整某个模型优先级

**6. 模型级路由分发**
系统的路由已经从 provider 级升级到 model 级。

也就是说，请求进来后，系统不是简单找一个 provider，而是：

- 先读取用户请求里的 `model`
- 在所有 provider 的 `provider_models` 中找匹配模型
- 过滤掉不可用模型
- 再按路由策略排序后选择候选

这样可以做到：

- 不支持该模型的 provider 不参与分发
- 同一 provider 下某个模型异常，不影响该 provider 的其他模型
- 真正实现“模型 -> 候选 provider 列表”的路由方式

**7. 路由模式**
当前系统已经支持以下路由模式：

- `manual`
- `failover`
- `weighted`
- `sticky`

具体能力如下：

`manual`
- 可以优先使用默认 provider
- 可配置是否允许 fallback

`failover`
- 先尝试默认 provider
- 失败后按优先级切换其他 provider

`weighted`
- 基于模型级权重做首选分发
- 再结合优先级排序回退

`sticky`
- 同一个用户 / 会话优先命中同一 provider
- 有助于保持会话体验稳定

**8. 内部强制指定 provider**
为了方便内部调试、灰度和人工分发，系统已经支持一个内部路由头：

- `X-Aotu-Provider-Id`

这个 header 允许管理员或内部调用方强制指定某个 provider 来处理请求，同时又不破坏主 OpenAI-compatible 协议。

普通用户不需要知道这个 header，也不会影响正常协议兼容。

**9. 健康检查系统**
健康检查已经从“只查 `/models`”升级成了两层结构。

`provider 级检查`
用于确认：

- base_url 可达
- 鉴权有效
- provider 整体在线

`model 级检查`
用于确认：

- 某个具体模型真的能调用
- 可选流式探测
- 可选视觉能力探测

模型探测默认使用最小测试请求，例如：

- `model`
- `messages: ping`
- `max_tokens: 1`
- `stream: false`

这样能更真实地判断“模型可不可用”，而不仅仅是接口活着。

**10. 手动与自动健康检查**
当前已经支持：

- 手动测试单个 provider
- 手动测试单个 provider 下的单个模型
- 手动测试全部 provider
- 手动测试全部 provider-model
- 自动巡检全部 provider / model

对应内部管理接口已经具备：

- `POST /api/providers/{id}/test`
- `POST /api/providers/{id}/models/{model_id}/test`
- `POST /api/providers/test-all`
- `GET /api/provider-models`
- `POST /api/provider-models/{id}/test`
- `POST /api/provider-models/test-all`

**11. 健康状态与熔断状态**
当前系统已经具备分层健康状态：

- `healthy`
- `degraded`
- `unhealthy`
- `unknown`

并且支持模型级熔断状态：

- `closed`
- `open`
- `half_open`

熔断逻辑已经具备：

- 连续失败达到阈值后 `open`
- 到恢复探测窗口后进入 `half_open`
- 探测成功回到 `closed`
- 探测失败重新 `open`

**12. 高级路由与自动调度**
系统已经不只是“静态转发”，而是做了动态调度。

当前调度会综合这些因子：

- 模型健康状态
- 模型优先级
- provider 优先级
- 模型权重
- 最近 5 分钟成功率
- 最近 5 分钟失败率
- 最近平均延迟
- 半开状态探测

并且具备：

- 自动降权
  - 最近 5 分钟失败率高的模型会被临时降权
  - 不会立刻彻底禁用
- 自动熔断
  - 连续失败达到阈值后进入 `open`
- 自动恢复
  - 到恢复时间后允许 `half_open` 探测
  - 恢复成功自动回到候选池

这意味着某条线路不稳定时，流量会自然向更稳定的副本迁移，而不是继续不断砸向坏线路。

**13. 请求日志与链路审计**
系统已经具备比较完整的请求日志能力。

当前日志会记录：

- 请求时间
- 请求路径
- 请求模型
- `requested_model`
- `resolved_provider_model_id`
- 是否 stream
- 是否包含图片
- 命中 provider
- 命中实际模型
- 状态码
- 耗时
- 错误摘要
- trace

trace 已经是结构化的，可以看到整个尝试链路，例如：

- 先尝试了哪个 provider
- 为什么失败
- 状态码是多少
- 延迟多少
- 最后成功命中了哪个 provider

**14. 错误分类能力**
当前系统已经能在日志和 trace 中区分常见错误类型，例如：

- `upstream_auth_error`
- `model_not_found`
- `rate_limited`
- `http_error`
- `request_rejected`
- `exception`
- `client_cancelled`
- `interrupted`

这样在排查问题时，可以更快知道到底是：

- 上游 key 错了
- 模型不存在
- 被限流了
- 上游 5xx
- 流式中断
- 客户端取消

**15. 可观测性与统计**
系统已经增加了聚合统计能力。

新增接口：

- `GET /api/logs/metrics`
- `GET /api/metrics/summary`

可以按最近 N 分钟、按 provider / model 聚合查看：

- 请求总数
- 失败数
- 失败率
- 平均延迟

这些能力已经可以支撑后续更强的自动调度和运营分析。

**当前最适合的使用边界**

这套系统现在最适合做的是：

- 统一管理多个 GPT 中转站
- 用统一 URL / API Key 对外服务
- 让用户像直接调用 GPT API 一样调用
- 在后台做路由、调度、探活、日志和熔断

它**不适合**做的边界仍然是：

- 原生 Claude / Gemini 协议适配
- 多家完全异构协议统一转换
- 复杂 agent 编排平台
- 聊天产品前台
