Get Started

第一次使用，建议这样开始

01

先把可用线路配出来

进入 /providers 添加中转站，补齐模型、价格、成功率、首 Token 时延和稳定性。这里决定系统到底有哪些线路可选。

02

再决定系统怎么选线路

进入 /settings 设置路由模式。你可以选择固定优先、自动回退、按权重分流或按会话粘滞。

03

给外部系统开通权限

进入 /api-keys 创建平台 API Key，绑定可访问的 provider，并设置 Token 额度、金额额度和余额。

04

先验证，再给业务系统上线

先在 /playground 验证链路，再让业务系统用平台 Key 接入。上线后主要看 /logs 和 /conversations。

Scenario Guide

按场景查看

我要新增一个 provider

到 /providers 新建渠道，录入模型和价格，然后先单独测试，再决定是否加入系统路由。

我要给业务系统开通调用

到 /api-keys 创建平台 Key，绑定授权 provider，设置额度和余额，再把平台 Key 交给业务系统。

我要控制成本和稳定性

回到 /providers 与 /settings，结合单价、成功率、首 Token 时间和稳定性评分来调整优先级和路由模式。

我要知道外部怎么接入

业务系统只需要 BASE_URL、平台 API Key 和标准请求参数；正式调用统一走 /v1/*。

我要排查一次失败请求

先查 /logs 看状态码和命中 provider，再看会话回放，最后回 API Key 详情确认是否额度或余额限制。

FAQ

高频问题

为什么外部系统不能直接指定 provider？

因为对外入口强调统一代理和稳定路由。外部只关心模型和结果，具体命中哪条线路由平台负责。

为什么我已经有上游 Key，还要再建平台 Key？

平台 Key 负责统一授权、额度、余额、账单和日志留痕，外部系统不应该直接持有上游渠道 Key。

为什么会被 429 拦截？

通常是 Token 额度耗尽、金额额度耗尽或余额不足，需要回 API Key 详情页确认哪一项触发限制。

新接入时优先用哪个接口？

优先使用标准 /v1/responses 或兼容的 /v1/chat/completions，不要走后台调试接口。

Billing

账务说明

Token 额度按总 Token 用量累计，到上限后停止调用
金额额度按总成本累计，到上限后同样停止调用
余额每次成功请求都会扣费，余额不足时直接拒绝
账单依据统一基于请求日志和模型单价自动计算
Troubleshooting

排查入口

401 / 403先查 API Key 是否错误、禁用或过期
429先查额度与余额
模型不可用先查 provider 配置、授权范围和路由策略
请求变慢先查 provider 降级、延迟和自动回退情况
