- 再由调度器每 `15s` 做一轮轻量回填，补旧数据和漏网数据
- 流式链路中仅额外收集用于计算的文本，不在 SSE 返回链路里做阻塞式 token 统计

结果：

- 用户首包和流式输出体验不受明显影响
- token 统计仍能较快补齐
- 兼顾了体验、准确性和资源消耗

**40. 流式场景下如果只保存截断文本，会导致 completion token 不准确**

问题现象：

- 项目原本为了控制日志体积，只保存受大小上限限制的 `response_text`
- 但如果直接拿这份截断文本去计算 token
- 长回复的 `completion_tokens` 会被低估

原因：

- “日志展示文本”与“token 计算文本”是两个不同目标
- 展示可以截断，但计算不能只靠截断内容

解决方法：

- 在流式处理中额外维护一份仅用于 token 补算的完整输出文本缓冲
- 该缓冲不参与用户响应，不用于页面大段展示
- 只在日志写入后交给后台 token 服务计算 `completion_tokens`

结果：

- 长流式回复的 token 统计更准确
- 同时保持原有日志大小控制策略不变

**41. 本地精确补算 Token 需要 tokenizer 依赖，项目原先缺失**

问题现象：

- 想在本地补算 token 时，项目环境里没有可用的 tokenizer 依赖
- 无法对 GPT 类模型做较精确的 token 计数

原因：

- 原项目依赖中没有引入 `tiktoken`
- 因而只能依赖上游 `usage`，不能独立完成本地补算

解决方法：

- 在项目依赖中新增：
  - `tiktoken`
- 并安装到该项目虚拟环境中
- 同时在代码里加入兼容处理：
  - 若依赖不可用，则不阻塞主程序启动
  - 若依赖可用，则启用本地补算能力

结果：

- 项目具备了本地 token 精确补算的基础能力
- 避免把 token 统计完全绑定在上游实现质量上

**42. “记录 Token 使用量”开关应同时控制后台补算行为**

问题现象：

- 如果管理员关闭了“记录 Token 使用量”
- 但后台补算任务仍继续运行
- 会造成“前台关闭、后台仍在算”的行为不一致

原因：

- token 补算属于 token 记录功能的一部分
- 若没有统一受设置项控制，就会产生功能语义不一致

解决方法：

- 将日志写入后的异步补算与定时回填任务，都统一绑定到：
  - `enable_token_logging`
- 当该开关关闭时：
  - 不再调度单条补算
  - 不再执行周期性回填

结果：

- 设置项行为与系统实际执行完全一致
- 管理员对 token 记录开关的预期更明确

# 2026-04-24 静态资源缓存导致页面脚本与模板不一致

- 问题现象：页面模板已更新，但浏览器仍加载旧版 `app.js` 或 `app.css`，出现按钮点击无效、列表不渲染、页面状态停留在旧占位内容等现象。
- 适用范围：所有依赖 `app/templates/base.html` 引入的前端静态资源变更。
- 原因：修改 `app/static/` 下资源后，未同步更新 `app/templates/base.html` 中资源 URL 的版本参数，浏览器继续命中缓存。
- 解决方法：每次修改会影响浏览器行为的静态资源后，同步更新 `base.html` 中对应资源的 `?v=` 参数，并让页面执行一次硬刷新或重新加载完整页面。
- 排查方式：当接口正常但页面交互失效时，优先检查浏览器实际加载的静态资源版本是否与当前代码一致。

# 2026-04-24 API Key 额度累计不能只依赖同步响应

- 问题现象：新增 API Key Token 额度统计后，部分请求日志会在异步 token 回填后才得到完整的 `prompt_tokens`、`completion_tokens`、`total_tokens`，导致 API Key 聚合用量与日志明细不一致。
- 适用范围：所有同时存在“请求聚合额度统计”和“日志异步补 token”的模块。
- 原因：如果只在主请求成功时同步累加 API Key 用量，而不处理后续日志 token 回填产生的差值，聚合统计会长期偏小。
- 解决方法：在日志 token 回填逻辑中同步计算与原值的 delta，并把差值补回到 API Key 聚合字段。
- 排查方式：当单密钥总用量与该密钥日志求和不一致时，优先检查异步 token 回填链路是否同步更新了聚合统计。

# 2026-04-24 外部 `/v1/*` 不得保留内部“指定中转站测试”能力

- 问题现象：对外 Bearer API Key 入口如果继续接受 `X-Aotu-Provider-Id` 之类的内部调试头，会把“指定某个具体中转站测试”的能力泄漏到外部调用链。
- 适用范围：所有同时存在“正式对外入口”和“后台内部调试入口”的代理项目。
- 原因：在拆分 `/v1/*` 和 `/api/playground/*` 时，只拆了鉴权和路径，没有同时清掉外部链路上残留的内部强制路由参数。
- 解决方法：外部 `/v1/*` 只允许按 API Key 路由策略执行，不再接受内部强制指定 provider 的头；“指定中转站测试”只保留在 `/api/playground/*`。
- 排查方式：当发现外部请求还能通过自定义头强制命中某个 provider 时，优先检查对外 router 是否仍保留了内部调试参数。

- 原因：后端过滤只排除了当前使用的 `health_check_provider` 与 `health_check_model`，没有兼容旧版本遗留的 `health_check` 类型。
- 解决方法：健康检查过滤统一按兼容表达式处理，至少覆盖旧值 `health_check` 和新值 `health_check_*`，不要只写当前枚举的精确排除列表。
- 排查方式：当“隐藏健康检查日志”仍能看到健康检查记录时，优先统计数据库里的 `log_type` 分布，确认是否存在历史枚举值未被过滤。

# 2026-04-24 路由评分不能混入健康检查日志

- 问题现象：同一 provider/model 明明真实业务请求成功，但自动健康检查失败后，`weighted`、`failover` 等模式的近期成功率和延迟会被明显拉低，导致路由判断失真。
- 适用范围：所有把近期请求日志用于动态路由评分、权重衰减、故障切换排序的模块。
- 原因：路由统计直接聚合 `request_logs` 最近窗口数据时，没有排除 `health_check`、`health_check_provider`、`health_check_model` 等健康检查日志。
- 解决方法：路由专用统计必须只统计真实代理流量日志，至少排除所有健康检查类型；若后续还有内部探测、回放、压测日志，也应统一排除。
- 排查方式：当路由评分与实际用户流量表现明显不一致时，优先按 `log_type` 分组检查最近窗口，确认是否被健康检查或内部流量污染。

# 2026-04-24 路由热路径不能反复做遗留数据同步和无索引聚合

- 问题现象：请求量上来后，路由入口响应时间会越来越依赖数据库扫描时间，`/v1/*` 首包延迟容易被放大。
- 适用范围：所有在请求主链路里同时执行 provider 列表读取、遗留数据修复、近期日志聚合的代理系统。
- 原因：路由候选构建热路径里重复触发 `ProviderService.sync_legacy_provider_models`，并对 `request_logs` 最近窗口做聚合；而日志表缺少围绕 `created_at`、`provider_id`、`requested_model`、`success` 的联合索引。
- 解决方法：把遗留数据同步迁出热路径，只在启动迁移或后台修复任务中执行；同时为路由统计所需字段补充索引，必要时把近期路由指标做增量汇总或缓存。
- 排查方式：当 `/v1/*` 没有明显上游延迟却仍变慢时，优先检查是否在热路径做了隐式修复逻辑，并查看路由统计 SQL 是否走索引。

# 2026-04-24 半开恢复状态必须持久化，不能只停留在当前 Session

- 问题现象：熔断恢复窗口到期后，同一个已打开线路会在多个请求里被重复当作“第一次探测”，出现并发重复探测和状态抖动。
- 适用范围：所有使用 `open -> half_open -> closed` 熔断恢复状态机的路由模块。
- 原因：候选筛选阶段把 `provider_model.circuit_state` 从 `open` 改成 `half_open` 后，没有立即持久化；请求结束后 Session 关闭，数据库仍保持 `open`。
- 解决方法：进入 `half_open` 时应原子化持久化，并记录探测时间；必要时增加单模型恢复锁或 compare-and-set 语义，避免并发请求同时抢到探测资格。
- 排查方式：当同一条已熔断线路在恢复窗口后被多个请求同时探测时，优先检查 `half_open` 状态是否真的落库。

# 2026-04-24 会话粘滞不能用候选列表取模代替一致性路由

- 问题现象：`sticky` 模式虽然对同一 key 可复现，但可能稳定命中较低优先级线路；候选列表一旦增减，原有粘滞映射也会整体漂移。
- 适用范围：所有需要“同会话尽量命中同一路线”且同时关注健康度、权重、扩缩容稳定性的路由场景。
- 原因：当前实现只是对排序后的候选列表做哈希取模，没有使用一致性哈希、rendezvous hashing 或带健康约束的会话亲和算法。
- 解决方法：若业务要求真实会话粘滞，应改为健康感知的一致性哈希方案，并把默认优先级、权重、摘除节点后的最小扰动一起纳入设计。
- 排查方式：当 `sticky` 模式出现“同一用户长期命中次优线路”或“节点增减后会话大面积漂移”时，优先检查是否仍在使用简单取模策略。


# 2026-04-24 成本与质量指标不能只埋在 trace_json 里

- 问题现象：需要按 provider / model 展示成功率、平均首 Token 时间、稳定性评分时，如果关键字段只存在 `request_logs.trace_json`，前端展示和后台聚合都会变得脆弱且成本高。
- 适用范围：所有需要把路由追踪信息进一步用于管理面板统计、排序依据、成本质量分析的模块。
- 原因：`trace_json` 适合保存完整尝试链路，但不适合承担高频聚合字段；像 `first_token_latency_ms` 这类指标如果没有一等字段，查询层只能反复解析 JSON。
- 解决方法：把需要长期复用的关键指标提升为明确字段落库，例如 `request_logs.first_token_latency_ms`；配置类信息如模型单价应直接存放在 `provider_models`，聚合展示再由服务层统一计算。
- 排查方式：当管理后台新增性能/成本统计需求时，优先检查当前指标是否仍只存在于 JSON 追踪字段中；若是，应先评估是否提升为结构化字段。

# 2026-04-24 推送 GitHub 时不能默认远程仓库为空

- 问题现象：本地仓库首次执行 `git push -u origin main` 时，若远程 `main` 已存在初始化提交或其他历史提交，会收到 `fetch first` 拒绝，无法直接推送。

- 解决方法：先检查本地是否有未提交改动并完成提交，再抓取远程分支状态；若远程已有历史提交，则先执行抓取/合并，按最小必要原则解决冲突并形成合并提交，之后再推送。
- 排查方式：当首次推送出现 `fetch first`、`non-fast-forward` 或远程已存在内容提示时，优先检查远程 `main` 是否已有初始化提交，并确认是否需要先合并远程历史。

- 解决方法：单独提供面向 Linux / 阿里云 ECS 的一键部署脚本，统一完成 `.venv`、`.env`、依赖安装、`systemd` 服务和 `nginx` 站点配置，避免让运维手工拼装部署步骤。
- 排查方式：当项目在服务器上能手动前台启动、但无法稳定公网访问或无法开机自启时，优先检查是否仍在复用本地 Windows 启动脚本，而没有使用 Linux 专用部署入口。


# 2026-04-24 金额账单不能只在主请求成功时同步扣费

- 问题现象：如果账单金额只在主请求返回时同步计算，遇到上游未返回 `usage`、流式回填 token、日志异步补 token 等场景，API Key 的累计消费、余额和账单流水会与最终日志不一致。
- 适用范围：所有同时存在“请求即时落日志”和“异步 token 回填 / 重算”的计费系统。

- 解决方法：账单系统必须复用同一条 `request_logs` 记录做重算；当 token 回填发生时，按成本 delta 同步更新 API Key 聚合消费、余额和账单流水，而不是只在首次成功响应时扣费一次。
- 排查方式：当“日志里的金额”“API Key 累计消费”“余额变化”“账单流水”四者不一致时，优先检查 token 回填链路是否同时触发了成本重算和账单 delta 同步。

- 排查方式：当新增 Session 登录态后应用在启动阶段而非请求阶段就报错时，优先检查 `requirements.txt` 是否已显式包含 `itsdangerous`。

# 2026-04-25 GitHub 上传必须按“先提交、再抓取合并、最后推送”顺序执行

- 问题现象：本地项目准备上传到 GitHub 时，若直接执行 `git push -u origin main`，容易遇到 `fetch first`、`non-fast-forward`、远程已有初始化提交或本地与远程不是同根历史等报错。

- 解决方法：固定按以下顺序执行：先运行 `git status -sb`、`git branch --show-current`、`git remote -v`、`git log --oneline -3` 确认状态；再执行 `git add .` 和 `git commit -m "<清晰提交信息>"`；若未配置远程则先 `git remote add origin <仓库地址>`；随后执行 `git fetch origin main`，按实际情况执行 `git merge origin/main` 或 `git merge origin/main --allow-unrelated-histories`；完成冲突解决并提交后，再执行 `git push -u origin main`。
- 排查方式：当推送被拒绝或提示远程已有内容时，优先检查是否漏做了本地提交、远程抓取和历史合并，而不是直接尝试强推覆盖。

- 问题现象：对外 `/v1/chat/completions`、`/v1/responses` 已经能成功调用，但后台“中转站测试”弹窗仍显示 `provider connectivity failed`、`模型通过 0/1` 或旧的 30 秒超时结果。
- 适用范围：所有同时提供正式代理入口与后台健康测试入口的 LLM 代理/中转项目。
- 原因：后台测试器单独使用了 `/models` 连通性探测、旧的文本 `ping` 或独立客户端逻辑，没有复用正式代理链路的真实入参、图片适配、超时和传输策略，导致测试口径与真实流量分裂。
- 解决方法：后台中转站测试必须改为复用正式代理同口径的探测逻辑，至少要按实际对外支持的 `/v1/chat/completions` 与 `/v1/responses` 路径逐项探测，并把失败原因按端点原样回显，而不是继续先测 `/models` 再笼统报“连通性失败”。
- 排查方式：当真实 API 调用成功，但后台测试页仍显示 `provider connectivity failed`、固定约 30 秒失败或模型结果与正式接口不一致时，优先检查测试器是否还在走独立的旧健康检查实现。

# 2026-04-26 Linux 部署健康检查不能在 `systemctl restart` 后只探测一次

- 问题现象：`start_aliyun.sh` 执行到本地健康检查时立即报 `curl: (7) Failed to connect to 127.0.0.1 port 8000`，但几秒后 `systemd` 状态和 `journalctl` 显示 `Uvicorn running on http://127.0.0.1:8000`，实际服务已启动成功。
- 适用范围：所有通过 `systemd` 重启 Python Web 服务后，立即用 `curl` 做本地或公网探活的部署脚本。
- 原因：`systemctl restart` 返回成功不代表应用已经完成启动监听；若脚本在服务完成应用启动前只探测一次，就会把“启动中”误判为“启动失败”。
- 解决方法：部署脚本中的健康检查必须改为有限次轮询重试，并在失败日志里区分“仍在等待启动”和“最终探活失败”；仅在重试耗尽后才输出 `systemctl status` 并退出。
- 排查方式：当脚本报本地探活失败，但 `systemctl status` 已显示 `active (running)`、且 `journalctl` 中随后出现 `Application startup complete` 时，优先检查探活是否执行过早，而不是先怀疑服务启动配置错误。

# 2026-04-26 自定义 HTTP 响应头不能直接写入中文渠道名

- 问题现象：代理接口在返回流式或非流式结果时，如果把中文中转站名直接写入 `X-Proxy-Provider-Name`，Starlette 会在构造响应头阶段抛出 `UnicodeEncodeError: 'latin-1' codec can't encode character ...`，导致请求直接失败。
- 适用范围：所有通过 FastAPI / Starlette / ASGI 自定义响应头回传渠道名、模型名、备注等非 ASCII 文本的接口。
- 原因：HTTP 响应头在 Starlette 初始化时会按 `latin-1` 编码，中文等超出 `latin-1` 范围的字符不能直接写入头部。
- 解决方法：需要回传中文文本时，必须先统一编码为可安全传输的 ASCII 形式，例如 UTF-8 百分号编码，并同步提供显式编码标识；浏览器或前端读取该头时再按约定解码展示。
- 排查方式：当接口日志显示业务逻辑已执行到返回阶段，但堆栈落在 `starlette.responses.init_headers` 且异常是 `latin-1` 编码失败时，优先检查最近新增的自定义响应头是否直接写入了中文或其他非 `latin-1` 字符。

# 2026-04-26 聚合告警落库前必须先把 `datetime` 转成 JSON 安全值

- 问题现象：打开 `/alerts` 告警中心时直接返回 `500 Internal Server Error`，服务层堆栈显示 `TypeError: Object of type datetime is not JSON serializable`。
- 适用范围：所有把聚合视图、告警快照、统计载荷等字典对象以 JSON 字符串形式持久化到数据库的 Python 后端服务。
- 原因：聚合 payload 经常混入 `datetime` 或 `date` 字段；若直接调用 `json.dumps()` 落库，标准 JSON 编码器不会自动处理这两类对象。
- 解决方法：在写入 `payload_json` 前，必须先做一层递归 JSON 安全转换，把 `datetime` / `date` 统一转成 ISO 8601 字符串，再执行序列化；不要假设上游 service 返回的聚合字典天然可直接 `json.dumps()`。
- 排查方式：当页面渲染前的 service 逻辑、事件持久化逻辑或缓存写入逻辑在 `json.dumps()` 处报 `datetime is not JSON serializable` 时，优先检查聚合 payload 中是否混入了时间字段。

# 2026-04-26 编辑现有聚合模型时，未显式操作的未绑定渠道不能被前端默认提交为新绑定

- 问题现象：在模型聚合列表中编辑某个已有模型并保存后，页面看起来像“生成了一个新模型”，实际是该模型被额外挂到了原本未绑定的渠道上。
- 适用范围：所有使用前端可编辑表格收集“是否绑定 / 是否启用 / 权重优先级”等布尔与数值组合配置的后台管理页面。
- 原因：编辑态表格里存在“当前未绑定”的候选行时，如果前端仅按最终 DOM 勾选状态组装提交载荷，而不区分“初始未绑定”和“用户本次显式勾选”，就可能把未实际操作的候选行误提交成新绑定。
- 解决方法：编辑态收集表格数据时，必须同时记录每一行的初始绑定状态和本次是否被用户显式改动；对初始未绑定且本次未被显式勾选的行，提交时必须保持 `bound=false`，避免误创建新绑定。
- 排查方式：当“编辑保存后多出一个模型”但主模型表并未新增记录、只是渠道绑定表多出一条关联时，优先检查前端编辑表单是否把未操作的候选渠道误当成新增绑定一起提交了。

# 2026-04-26 后台测试展示层若不展开正式代理双端点明细，会继续放大“测试失败但真实调用成功”的误判

- 问题现象：后端测试逻辑已经按正式代理口径同时探测 `/v1/chat/completions` 与 `/v1/responses`，但后台弹窗或批量测试卡片仍只显示单条模型总结果，用户看不到到底是哪一个正式端点成功、哪一个失败。
- 适用范围：所有把多端点探测结果汇总成 `model_results[].endpoint_results[]` 返回给前端的代理/中转后台项目。
- 原因：展示层只消费模型级 `success/message` 聚合字段，没有继续展开 `endpoint_results`，导致定位信息在前端被截断，测试口径虽然已修正，观察口径却仍停留在旧的一维结果。
- 解决方法：前端测试弹窗和批量测试结果必须把每个模型下的正式端点明细一起展示，至少回显端点名、成功/失败、状态码、耗时和返回消息；模型级结构也应显式带上 `model_name`，避免列表展示退化成匿名结果。
- 排查方式：当后台测试页提示“模型失败”但真实外部调用至少一个正式端点成功时，优先检查前端是否遗漏了 `endpoint_results` 渲染，而不是先怀疑后端探测本身仍在走旧逻辑。
