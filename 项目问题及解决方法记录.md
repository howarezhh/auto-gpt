- 再由调度器每 `15s` 做一轮轻量回填，补旧数据和漏网数据
- 流式链路中仅额外收集用于计算的文本，不在 SSE 返回链路里做阻塞式 token 统计

结果：

- 用户首包和流式输出体验不受明显影响
- token 统计仍能较快补齐
- 兼顾了体验、准确性和资源消耗

**40. 流式场景下如果只保存截断文本，会导致 completion token 不准确**

问题现象：

- 项目原本为了控制日志体积，只保存受大小上限限制的 `response_text`
- 但如果直接拿这份截断文本去计算 token
- 长回复的 `completion_tokens` 会被低估

原因：

- “日志展示文本”与“token 计算文本”是两个不同目标
- 展示可以截断，但计算不能只靠截断内容

解决方法：

- 在流式处理中额外维护一份仅用于 token 补算的完整输出文本缓冲
- 该缓冲不参与用户响应，不用于页面大段展示
- 只在日志写入后交给后台 token 服务计算 `completion_tokens`

结果：

- 长流式回复的 token 统计更准确
- 同时保持原有日志大小控制策略不变

**41. 本地精确补算 Token 需要 tokenizer 依赖，项目原先缺失**

问题现象：

- 想在本地补算 token 时，项目环境里没有可用的 tokenizer 依赖
- 无法对 GPT 类模型做较精确的 token 计数

原因：

- 原项目依赖中没有引入 `tiktoken`
- 因而只能依赖上游 `usage`，不能独立完成本地补算

解决方法：

- 在项目依赖中新增：
  - `tiktoken`
- 并安装到该项目虚拟环境中
- 同时在代码里加入兼容处理：
  - 若依赖不可用，则不阻塞主程序启动
  - 若依赖可用，则启用本地补算能力

结果：

- 项目具备了本地 token 精确补算的基础能力
- 避免把 token 统计完全绑定在上游实现质量上

**42. “记录 Token 使用量”开关应同时控制后台补算行为**

问题现象：

- 如果管理员关闭了“记录 Token 使用量”
- 但后台补算任务仍继续运行
- 会造成“前台关闭、后台仍在算”的行为不一致

原因：

- token 补算属于 token 记录功能的一部分
- 若没有统一受设置项控制，就会产生功能语义不一致

解决方法：

- 将日志写入后的异步补算与定时回填任务，都统一绑定到：
  - `enable_token_logging`
- 当该开关关闭时：
  - 不再调度单条补算
  - 不再执行周期性回填

结果：

- 设置项行为与系统实际执行完全一致
- 管理员对 token 记录开关的预期更明确

# 2026-04-24 静态资源缓存导致页面脚本与模板不一致

- 问题现象：页面模板已更新，但浏览器仍加载旧版 `app.js` 或 `app.css`，出现按钮点击无效、列表不渲染、页面状态停留在旧占位内容等现象。
- 适用范围：所有依赖 `app/templates/base.html` 引入的前端静态资源变更。
- 原因：修改 `app/static/` 下资源后，未同步更新 `app/templates/base.html` 中资源 URL 的版本参数，浏览器继续命中缓存。
- 解决方法：每次修改会影响浏览器行为的静态资源后，同步更新 `base.html` 中对应资源的 `?v=` 参数，并让页面执行一次硬刷新或重新加载完整页面。
- 排查方式：当接口正常但页面交互失效时，优先检查浏览器实际加载的静态资源版本是否与当前代码一致。

# 2026-04-24 API Key 额度累计不能只依赖同步响应

- 问题现象：新增 API Key Token 额度统计后，部分请求日志会在异步 token 回填后才得到完整的 `prompt_tokens`、`completion_tokens`、`total_tokens`，导致 API Key 聚合用量与日志明细不一致。
- 适用范围：所有同时存在“请求聚合额度统计”和“日志异步补 token”的模块。
- 原因：如果只在主请求成功时同步累加 API Key 用量，而不处理后续日志 token 回填产生的差值，聚合统计会长期偏小。
- 解决方法：在日志 token 回填逻辑中同步计算与原值的 delta，并把差值补回到 API Key 聚合字段。
- 排查方式：当单密钥总用量与该密钥日志求和不一致时，优先检查异步 token 回填链路是否同步更新了聚合统计。

# 2026-04-24 外部 `/v1/*` 不得保留内部“指定中转站测试”能力

- 问题现象：对外 Bearer API Key 入口如果继续接受 `X-Aotu-Provider-Id` 之类的内部调试头，会把“指定某个具体中转站测试”的能力泄漏到外部调用链。
- 适用范围：所有同时存在“正式对外入口”和“后台内部调试入口”的代理项目。
- 原因：在拆分 `/v1/*` 和 `/api/playground/*` 时，只拆了鉴权和路径，没有同时清掉外部链路上残留的内部强制路由参数。
- 解决方法：外部 `/v1/*` 只允许按 API Key 路由策略执行，不再接受内部强制指定 provider 的头；“指定中转站测试”只保留在 `/api/playground/*`。
- 排查方式：当发现外部请求还能通过自定义头强制命中某个 provider 时，优先检查对外 router 是否仍保留了内部调试参数。

- 原因：后端过滤只排除了当前使用的 `health_check_provider` 与 `health_check_model`，没有兼容旧版本遗留的 `health_check` 类型。
- 解决方法：健康检查过滤统一按兼容表达式处理，至少覆盖旧值 `health_check` 和新值 `health_check_*`，不要只写当前枚举的精确排除列表。
- 排查方式：当“隐藏健康检查日志”仍能看到健康检查记录时，优先统计数据库里的 `log_type` 分布，确认是否存在历史枚举值未被过滤。

# 2026-04-24 路由评分不能混入健康检查日志

- 问题现象：同一 provider/model 明明真实业务请求成功，但自动健康检查失败后，`weighted`、`failover` 等模式的近期成功率和延迟会被明显拉低，导致路由判断失真。
- 适用范围：所有把近期请求日志用于动态路由评分、权重衰减、故障切换排序的模块。
- 原因：路由统计直接聚合 `request_logs` 最近窗口数据时，没有排除 `health_check`、`health_check_provider`、`health_check_model` 等健康检查日志。
- 解决方法：路由专用统计必须只统计真实代理流量日志，至少排除所有健康检查类型；若后续还有内部探测、回放、压测日志，也应统一排除。
- 排查方式：当路由评分与实际用户流量表现明显不一致时，优先按 `log_type` 分组检查最近窗口，确认是否被健康检查或内部流量污染。

# 2026-04-24 路由热路径不能反复做遗留数据同步和无索引聚合

- 问题现象：请求量上来后，路由入口响应时间会越来越依赖数据库扫描时间，`/v1/*` 首包延迟容易被放大。
- 适用范围：所有在请求主链路里同时执行 provider 列表读取、遗留数据修复、近期日志聚合的代理系统。
- 原因：路由候选构建热路径里重复触发 `ProviderService.sync_legacy_provider_models`，并对 `request_logs` 最近窗口做聚合；而日志表缺少围绕 `created_at`、`provider_id`、`requested_model`、`success` 的联合索引。
- 解决方法：把遗留数据同步迁出热路径，只在启动迁移或后台修复任务中执行；同时为路由统计所需字段补充索引，必要时把近期路由指标做增量汇总或缓存。
- 排查方式：当 `/v1/*` 没有明显上游延迟却仍变慢时，优先检查是否在热路径做了隐式修复逻辑，并查看路由统计 SQL 是否走索引。

# 2026-04-24 半开恢复状态必须持久化，不能只停留在当前 Session

- 问题现象：熔断恢复窗口到期后，同一个已打开线路会在多个请求里被重复当作“第一次探测”，出现并发重复探测和状态抖动。
- 适用范围：所有使用 `open -> half_open -> closed` 熔断恢复状态机的路由模块。
- 原因：候选筛选阶段把 `provider_model.circuit_state` 从 `open` 改成 `half_open` 后，没有立即持久化；请求结束后 Session 关闭，数据库仍保持 `open`。
- 解决方法：进入 `half_open` 时应原子化持久化，并记录探测时间；必要时增加单模型恢复锁或 compare-and-set 语义，避免并发请求同时抢到探测资格。
- 排查方式：当同一条已熔断线路在恢复窗口后被多个请求同时探测时，优先检查 `half_open` 状态是否真的落库。

# 2026-04-24 会话粘滞不能用候选列表取模代替一致性路由

- 问题现象：`sticky` 模式虽然对同一 key 可复现，但可能稳定命中较低优先级线路；候选列表一旦增减，原有粘滞映射也会整体漂移。
- 适用范围：所有需要“同会话尽量命中同一路线”且同时关注健康度、权重、扩缩容稳定性的路由场景。
- 原因：当前实现只是对排序后的候选列表做哈希取模，没有使用一致性哈希、rendezvous hashing 或带健康约束的会话亲和算法。
- 解决方法：若业务要求真实会话粘滞，应改为健康感知的一致性哈希方案，并把默认优先级、权重、摘除节点后的最小扰动一起纳入设计。
- 排查方式：当 `sticky` 模式出现“同一用户长期命中次优线路”或“节点增减后会话大面积漂移”时，优先检查是否仍在使用简单取模策略。
